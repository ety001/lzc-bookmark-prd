**评审者:** Composer 2  
**评审日期:** 2026-04-16

**文档路径:** `PRD-v4.md`

## 1. 总体结论

v4 在 v3 基础上**逐项落实**了此前评审中最关键的缺口：**每条变更全局版本 +1**、**增量拉取谓词 `to_version > since_version`**、**先推后拉与 `since_version=last_known_version` 的衔接**（§13.3）、**部分冲突时的客户端状态机**、**同 `parent_guid` 才合并重复创建**、**`reindex` 与归一化阈值**、**只读 Key 禁止写接口**、**设备表改为 `api_key_id`**、**本地备份默认关闭**、**回滚加锁与并发**、**状态机改为 PUSHING/PULLING** 等。

当前文档已达到**可直接驱动测试用例编写**的粒度。剩余问题以 **文案自相矛盾（变更日志保留条件）**、**同批 create 的父节点引用**、**Chrome 三根与服务端 `root` 的映射**、以及少数 **边界语义** 为主。

**整体评级:** 优秀（较 v3 明显提升可执行性）  
**建议状态:** 可进入开发；发布前建议修正 §4.2 保留条件表述并补一节「首批 / 大批 create 的 GUID 分配协议」。

## 3. 问题与建议（按优先级）

### P0 — 建议尽快修正文档或实现前定稿

#### 3.1 §4.2「变更日志保留条件」逻辑矛盾

§4.2 当前写：

- 「保留条件：`COUNT(change_log) <= 1000` **OR** `age <= 7 days`（满足任一即保留）」
- 紧接着「溢出行为：截断旧记录，保留最近1000 条**且**不超过 7 天」

二者语义冲突：**OR** 会导致「只要未满 7 天即永不因条数截断」之类误读；而「1000 条且7 天」实为 **AND** 型约束。

**建议:** 改为与溢出行为一致的单一表述，例如：

- 「日志仅在同时满足 **条数 ≤1000** 且 **最早一条年龄 ≤7 天** 时视为未溢出；否则截断并更新 baseline。」或明确分两句：**条数上限**与**时间上限**各自触发截断的规则。

**涉及段落:** §4.2（变更日志溢出策略）。

#### 3.2 同一次 `sync` 内多条 `create` 的 `parent_guid` 引用

§8.2 要求拓扑有序、§6.3.2 要求父先于子，但子节点的 `parent_guid` 在服务端尚未存在时，客户端若填 **服务端 GUID**，首批上传无法预知。

**缺口:** 未规定下列之一：

- 客户端对「同批新建父节点」使用 **临时 ID**，服务端按序解析并替换为正式 GUID；或  
- 分轮提交：每批仅含「父已存在 GUID」的 creates；或  
- `parent_local_id` + 服务端根据映射表解析（仅请求体、不进规范树）。

**建议:** 在 §6.3.2 或 §8.2 增加**最短一节**「同批 create 的父引用编码」，避免实现阶段各自发挥导致互不兼容。

**涉及段落:** §6.3.2、§8.2、§4.1 `PendingChange`。

### P1 — 强烈建议补充，减少联调与用户歧义

#### 3.3 服务端 `root` 与 Chrome 三根（书签栏 / 其他 / 移动）的映射

§8.3 已承认 Chrome 侧存在多个系统根，但规范树以 `guid: "root"` 为顶层。需固定：

- 规范树 `root` 的子节点与 `BOOKMARK_BAR`、`OTHER`、`MOBILE`（若存在）的**一一对应**；  
- 新设备 BFS `create` 时 **禁止**误在错误根下重建（否则多设备树形不一致）。

**建议:** 在 §8.3 增加小表「`parent_guid` → Chrome `parentId`规则」或「层级1 固定三条子树」。

**涉及段落:** §8.3、§6.3.1 示例 `children`。

#### 3.4 `409 E003` 与「HTTP 200 + conflicts」的边界

§6.1 保留 E003（乐观锁 / 版本冲突）。§6.3.2 又以200 + `conflicts` 处理部分冲突。

**建议:** 用两三句话定义：**何时返回 409**（例如 `base_version` 与服务器当前头版本严重不一致、必须重拉或全量；或重复 `client_request_id` 以外的非法乐观锁），**何时仅返回 200 + conflicts**（可部分应用的业务冲突）。否则客户端错误处理易混。

**涉及段落:** §6.1、§6.3.2。

#### 3.5 同文件夹「同名同 URL」自动合并

§4.3 仍规定 **同 `parent_guid` 则合并为同一 GUID**。用户偶发会需要两条完全相同的快捷方式（少见但合法）。

**建议:** 标注为 **v1 简化策略** 或改为「默认合并 + 高级选项保留两条」（若不做则接受为已知限制并在 FAQ 说明）。

**涉及段落:** §4.3。

#### 3.6 `reindex` 不计入 1000 条配额时的存储与版本

§4.4 / §7.3 写明 `reindex` 不占1000 配额，但若仍 **每条 +1 版本** 且持久化大量 `reindex` 记录，存储仍增长。

**建议:** 一句即可：是否**折叠连续 reindex**、或仅保留最新树状态 + 可选审计。非阻塞，利于运维估算。

**涉及段落:** §4.4、§7.3。

#### 3.7 回滚变更「完整树差异」与单条 `+1` 版本

§6.5 / §7.3 写 rollback 为一条变更、版本 +1，同时 §7.3 称含「完整树差异」。若单条日志体积极大，可能影响 DB 与响应。

**建议:** 注明实现可选用「指向快照 version + 差分摘要」或接受「大 BLOB」并设上限，与 §6.7 数据上限协调。

**涉及段落:** §6.5、§7.3、§7.6。

### P2 — 文档 polish

#### 3.8 §13.3 与无本地变更时的拉取

§13.3 示例覆盖「推送后拉取」。若本地无队列变更，是否仍 **GET** `since_version=last_known`以拉取他机变更，建议在 §3.4 或 §8.1 用一句补全（通常：**每轮闹钟/唤醒均拉取**）。

**涉及段落:** §3.4、§8.1、§13.3。

#### 3.9 术语「v1」

§9.2、§7.1 等「v1 暂不加密」「tenant v1 为空」与 **PRD v4** 并存，新读者易混。

**建议:** 改为「当前里程碑 / 书签同步首版能力」等表述。

**涉及段落:** §9.2、§7.1、§3.3。

#### 3.10 验收 §12.2「10 设备全量拉取」

与默认个人2–5 台场景略超前，作为压力项合理；可注明 **可选压测** 以免阻塞 MVP 发布。

**涉及段落:** §12.2。

## 5. 结语
