# 懒猫书签同步系统 PRD v4.0 - 审计报告

**审计者:** kimi-k2.5  
**审计时间:** 2026-04-16  
**PRD 版本:** v4.0  

---

## 总体评价

PRD v4.0 相比 v3.0 有**精细化改进**，核心机制更加明确和严谨。主要改进包括：
- ✅ 版本号粒度明确（每条变更 +1，非每次 sync +1）
- ✅ 增量拉取谓词精确化（`to_version > since_version`）
- ✅ 部分冲突处理规则补充（部分应用 vs 全部冲突）
- ✅ 新设备拉取 BFS 步骤细化
- ✅ 创建冲突合并范围缩小（同 parent_guid 才合并）
- ✅ index_weight 归一化行为明确（生成 reindex 变更）
- ✅ 本地备份默认关闭（用户体验优化）
- ✅ 只读 API Key 禁止写操作（权限控制强化）

文档已达到**生产级 PRD** 标准，细节充分，可直接指导开发。

### 4. 核心设计决策 ✅

#### 4.1 书签身份模型 ✅

**新增细节:**
- 首次上传顺序：**父节点先于子节点（拓扑有序）**
- 新设备拉取：**按 BFS 顺序创建书签**

**BFS 创建顺序示例清晰:**
```
Level 0: root
Level 1: 书签栏、其他书签
Level 2: 书签栏下的文件夹 A、文件夹 B
...
```

**建议:**
- BFS 顺序确保父节点先于子节点创建，避免 parent_id 引用不存在的问题
- Chrome 的 3 个根节点（书签栏、其他书签、移动设备）需要特殊处理

#### 4.2 增量同步模型 ✅

**版本号语义明确化（v4 核心改进）:**
```
每条变更（Change）对应全局版本号 +1
一次 sync 请求提交 N 条变更，全局版本号 +N
```

**增量拉取谓词精确化:**
```
服务端返回：所有满足 to_version > since_version 的变更
排序：按 to_version 升序
```

**示例清晰:**
```
since_version = 120
变更日志：
  - Change A: to_version=120  ← 不返回（to_version 不大于 120）
  - Change B: to_version=121  ← 返回
  - Change C: to_version=122  ← 返回
```

**变更日志溢出策略明确:**
- 保留条件：`COUNT <= 1000` **OR** `age <= 7 days`（满足任一即保留）
- 触发溢出：超过 1000 条 **且** 超过 7 天

**建议:**
- 明确"截断旧记录"的具体行为（删除还是归档？）

#### 4.3 冲突检测与解决 ✅

**创建冲突范围缩小（v4 改进）:**
```
v3: 同名 + 同 URL → 合并
v4: 同名 + 同 URL + 同 parent_guid → 合并
     同名 + 同 URL + 不同 parent_guid → 保留两个书签
```

**设计合理:** 用户可能有意将相同书签放在不同文件夹。

**部分冲突处理（v4 新增）:**

场景：推送 10 条变更，8 条无冲突，2 条有冲突。

```json
{
  "new_version": 128,        // 8 条变更应用，版本号 +8
  "applied_changes": [...],  // 8 条已应用
  "conflicts": [...]         // 2 条挂起
}
```

**客户端行为明确:**
1. 更新 `last_known_version = new_version`（128）
2. 从离线队列移除已应用的 8 条
3. 保留 2 条冲突变更，标记为"待解决"
4. 下次推送仅推送冲突变更 + 新增变更

**设计优秀:** 非阻塞冲突处理，用户可继续同步其他变更。

#### 4.4 排序权重设计 ✅

**归一化策略明确（v4）:**
- 触发条件：最小间隔 < 1e-10
- 归一化行为：重新分配权重为 1.0, 2.0, 3.0...
- **生成 `reindex` 类型变更记录**（特殊类型，不计入 1000 条配额）

**设计合理:** 避免权重空间耗尽，同时不占用正常变更日志配额。

### 5. 功能需求 ✅

#### 5.1 懒猫微服 ✅

**权限控制强化:**
- 只读 API Key 禁止调用 POST /sync、POST /conflicts/resolve、POST /rollback
- 返回 E002（权限不足）

**物理删除条件明确:**
- `is_deleted=true` AND `deleted_at > 90 days` AND **无快照引用**

**建议:**
- 明确"无快照引用"的检查范围（最近 30 天的快照？）

#### 5.2 Chrome 浏览器插件 ✅

**本地备份默认关闭（v4 变更）:**
- v3: 默认开启
- v4: 默认关闭，用户可开启

**设计合理:** 避免频繁导出打扰用户。

**Service Worker 休眠降级简化（v4）:**
- v3: 全量 diff 检测
- v4: 唤醒时直接 GET /bookmarks?since_version=N 拉取服务端变更

**设计简化合理:** 减少本地计算，依赖服务端状态。

### 6. API 设计 ✅

#### 6.1 通用规范 ✅

无变更。

#### 6.2 认证接口 ✅

**说明:** 非书签接口的 `version` 字段为 `null`。

#### 6.3 书签接口 ✅

**GET /api/v1/bookmarks** ✅

**增量拉取谓词形式化:**
```
服务端返回：所有满足 to_version > since_version 的变更
排序：按 to_version 升序
```

**POST /api/v1/bookmarks/sync** ✅

**幂等行为明确（v4）:**
```
检测到重复的 client_request_id：
1. 不重复应用变更（幂等）
2. 返回当前最新状态（可能包含其他设备的变更）
3. version 字段返回当前全局最新版本
```

**示例清晰:**
```
首次请求：client_request_id="req_001", base_version=120
  → 应用变更，返回 version=125

重试请求：client_request_id="req_001", base_version=120（相同）
  → 不应用变更，返回 version=128（其他设备可能已提交变更）
```

**客户端状态更新规则（v4 新增）:**
```
部分冲突：更新 last_known_version = new_version
全部冲突：不更新 last_known_version
```

#### 6.4 设备接口 ✅

**API Key 与 Device 关系明确（v4）:**
- Device 表存储 `api_key_id`（外键）
- 限流维度：`api_key_id + device_id` 组合

#### 6.5 历史接口 ✅

**回滚原子性明确（v4）:**
```
1. 获取全局写锁（或乐观锁）
2. 加载指定版本快照
3. 生成 "rollback" 类型变更
4. 追加到 change_log，版本号 +1
5. 释放锁

并发推送：等待锁释放（最长 5 秒），超时返回 E007
```

**设计合理:** 确保回滚操作的原子性。

#### 6.6 限流策略 ✅

**新增兜底上限:**
- 单 API Key 总限流：**每分钟 200 次**

**设计合理:** 防止单个 API Key 过度使用。

---

### 7. 数据模型 ✅

**Change 类型扩展:**
```typescript
type: "create" | "update" | "delete" | "move" | "reindex" | "rollback"
```

**特殊类型:**
- `reindex`: index_weight 归一化生成，不计入配额
- `rollback`: 回滚操作生成，包含完整树差异

**Device 模型改进:**
```typescript
// v3
api_key_hash: string

// v4
api_key_id: string  // 外键，一个 Key 可对应多个 Device
```

**Snapshot 存储估算（v4 补充）:**
- 5000 个书签：约 500KB（原始 JSON）→ 100-150KB（gzip）
- 500 个快照：约 50-75MB

---

### 8. 同步逻辑 ✅

**完整同步流程图改进:**
- 明确部分冲突 vs 全部冲突的分支
- 明确 `last_known_version` 更新规则

**新设备首次同步流程细化（v4）:**
```
1. GET /bookmarks（无 since_version）
2. 返回完整规范树 + is_baseline=true
3. 按 BFS 顺序创建本地书签：
   a. 创建根文件夹
   b. 遍历规范树，对每个节点：
      - chrome.bookmarks.create({parentId, title, url})
      - Chrome 返回 local_id
      - 建立映射：{local_id ↔ guid}
   c. 存储映射表
```

**BFS 顺序确保:**
- 父节点先于子节点创建
- 避免 parent_id 引用不存在的问题

**错误恢复场景补充（v4）:**
- 回滚操作中收到 POST /sync：等待锁释放（最长 5 秒），超时返回 E007
