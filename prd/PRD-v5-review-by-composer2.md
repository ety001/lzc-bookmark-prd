## 1. 总体结论

v5 基本完成了 v4/v3 评审遗留的核心落地：

- **变更日志截断**从 OR/AND 歧义走向可执行规则
- **同批 create 的父引用**补齐为 `parent_local_id` 协议
- **Chrome 三根映射**明确为 `root_bar/root_other/root_mobile` 固定节点
- **冲突时间窗口**限定为 UI 提示，不再影响判定
- **reindex 变更格式**补全
- **GET /bookmarks 增量分页**补充 `limit`

整体已达到「可以按文档直接开工并写自动化用例」的成熟度。剩余问题主要集中在：**410/E008 的触发时机与全量返回策略存在自相矛盾**、以及少量实现细节需要再钉死以避免客户端/服务端各自解读。

**整体评级:** 优秀（接近定稿）  
**建议状态:** 修复 P0 后可进入开发；P1 可在开发早期同步澄清。

## 2. 关键问题与建议（按优先级）

### P0 — 必须在开发启动前定稿

#### 2.1 `since_version < baseline_version` 时到底是 200 全量还是 410？（文档矛盾）

- §6.3.1 写：**无 since_version 或 since_version < baseline_version** → **HTTP 200** 返回全量树。
- §4.2（溢出策略）与 §8.6 写：变更日志溢出/落后 → **HTTP 410 + E008**，客户端全量重拉。

这两种行为只能选一，否则客户端无法稳定实现「落后设备恢复」逻辑。

**建议（二选一）：**

- **方案 A（推荐，少一次往返）**：`since_version < baseline_version` 直接 **200 全量**（带 baseline/current 细节），删除或收紧 410 的场景；
- **方案 B（严格语义）**：`since_version < baseline_version` 必须 **410**，客户端改为发「无 since_version」再取全量；则需把 §6.3.1 的全量条件改为「无 since_version」。

同时建议在全量响应中返回 `current_version`（现在用顶层 `version` 表示）并在文字里写清。

---

### P1 — 强烈建议补充，减少联调争议

#### 2.2 `parent_guid` 与 `parent_local_id` 同时出现时的优先级

在 `POST /sync` 的 create 变更示例里同时有：

- `parent_local_id`（用于同批创建父子关系）
- `data.parent_guid`（示例填了 `root_bar`）

**建议明确优先级：**

- 若父节点为「本批新建」，必须以 `parent_local_id` 解析父 GUID，忽略/覆盖 `data.parent_guid`；
- 若父节点已存在于规范树，则 `parent_local_id` 应为 `null` 且以 `data.parent_guid` 为准；
- 两者同时存在且不一致时返回 E005。

否则实现容易出现「子节点被挂到错误父目录」的隐性数据损坏。

#### 2.3 增量分页续页的安全性：`since_version = last_to_version` 是否会漏/重

当前谓词为 `to_version > since_version`（正确），续页用最后一条的 `to_version` 作为下一次 `since_version`（也正确）。

**建议补一句：** 服务端按 `to_version` 升序返回，且同一 `to_version` 不会对应多条变更（因为每条变更 +1 版本）。这样可证明不重不漏。

#### 2.4 `GET /bookmarks 响应大小 10MB → E007，建议全量同步` 的语义

§6.7 写到响应大小 >10MB 返回 E007 并「建议全量同步」，但此时如果已经是全量请求/或增量超大，客户端该做什么仍不明确。

**建议：**

- 若增量变更过大：返回一个可区分错误（仍可用 E007，但 `details.reason="response_too_large"`），客户端转为全量拉取；
- 若全量仍过大：需指向更强的降级（例如分片全量/按根分开拉取/分页树）。至少给出明确下一步。

#### 2.5 首次全量分批提交中的“覆盖语义”表述过于抽象

§8.2 分批策略最后一句「书签是覆盖语义，不会丢失数据」容易被误解（是指服务端把首同步当作一次快照覆盖？还是指 create 幂等？）。

**建议:** 具体化为：

- 每个 batch 的 create 仅追加缺失节点，不会删除服务端已有节点；或
- 首次同步是“重置式导入”（会先清空再写），并声明该模式只允许空库/管理员确认。

两者对安全性与可恢复性影响很大，必须明确。

#### 2.6 `rollback` 变更的客户端应用方式

PRD 写 rollback 变更包含 `rollback_to_version`、`affected_guids`，并通过「快照 + 差分」表达完整树差异。

**建议:** 补一句客户端策略：收到 rollback 相关变更后，是否需要强制全量重拉？（通常建议：直接全量重拉到新版本，避免客户端实现复杂差分回放）。

### P2 — 可优化但不阻塞

#### 2.7 三根中 `root_mobile` 的可选性

Chrome 的 Mobile 根在桌面端可能存在但为空，或在某些实现中不可用。PRD 已写“可选”，建议在 API/规范树里明确：

- `root_mobile` 节点是否总是存在（即便空），还是可缺省；
- 客户端缺省时的处理（跳过/创建占位）。

#### 2.8 冲突判定完全由 `from_version` 决定的副作用

PRD 已明确时间窗口仅 UI 提示，这是好的。建议补一个“预期行为”说明：即使两个变更在 1 小时后才到达，只要都基于同一 `from_version`，也会产生冲突（避免产品误解）。

## 3. 一致性快速核对

- **版本语义**：每条变更 +1，`to_version > since_version`，与分页续页逻辑一致。  
- **部分冲突**：`new_version` 表示已应用部分推进后的版本，顶层 `version` 表示全局头版本，客户端更新规则写清。  
- **权限矩阵**：只读/读写覆盖到设备撤销与回滚，且与 E002 对齐。  
- **三根映射**：规范树第一层固定三节点，且新设备拉取“不创建第一层”。  

唯一需要立即消歧的是 **P0（410 vs 200 全量）**，否则落后设备恢复实现会分叉。

---

## 4. 结语

v5 已把同步协议的大部分“坑”补齐到可实现水平。把 **410/E008 的触发时机**与 **全量返回策略**统一后，这份 PRD 可以作为服务端/插件端实现与验收测试的单一事实来源。
