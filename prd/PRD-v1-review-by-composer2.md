## 3. 关键问题与缺口（按优先级）

### P0 — 必须在 v1 规格中写清

#### 3.1 Chrome `id` 无法作为跨设备稳定主键

PRD 在数据模型中把 `id` 注释为「Chrome 书签 ID」。在 Chrome 中，书签节点的 `id` 是**当前配置内**由浏览器分配的稳定标识，但**不同设备 / 不同配置文件之间同一「逻辑书签」的 `id` 通常不一致**。

**影响:** 若服务端与冲突检测以 `node_id` = Chrome `id` 为主键，多设备同步会出现「无法对应同一书签」「误报冲突或漏检」等问题。

**建议:**

- 引入与服务端一致的**逻辑 ID**（例如 UUID）；或
- 明确 **稳定键（stable key）** 规则：如 `(normalized_url)` 对书签、`parent_logical_id + folder_title + index` 对文件夹等，并说明文件夹改名、移动时的重算方式；或
- 采用 **操作日志 + 树 diff** 模型，在客户端生成不依赖对端 Chrome `id` 的变更描述。

**PRD 涉及段落:** §6.1、§7.2、§6.3 `node_id`、§6.4 `node_id`。

#### 3.2 「增量同步」的双重含义未区分

文档同时出现：

- `GET /api/v1/bookmarks?since=<timestamp>`（按时间过滤），与
- `POST ... "changes": [...]`（变更列表），以及全局 `version`。

**缺口:** `since` 与 `version` 的关系、增量返回的是「变更列表」还是「自某版本以来的树补丁」、客户端如何与 `base_version` 对齐，均未定义。

**建议:** 明确 v1 只选一种主方案（例如 **仅版本号 + 变更日志**，或 **快照 + 客户端 diff**），并在 API 中删掉或合并冗余参数，避免实现分叉。

**PRD 涉及段落:** §4.1.3、§5.2、§6.2、§7.1。

#### 3.3 架构图中的 WebSocket 与正文 API 不一致

§3 架构图标注插件与懒猫之间为 HTTP与 WebSocket；§5 API 设计**全部为 REST**，未说明 WebSocket 的用途（推送版本更新、冲突通知、管理端实时刷新等）。

**建议:** 二选一写进 PRD：（1）v1 **仅 HTTP（+ 轮询 / 长轮询）**，图中删除或注明「未来」；  
（2）补充 WebSocket **事件模型**（订阅主题、鉴权方式、与 `version` 的联动）。

**PRD 涉及段落:** §3 图、§5全文。

### P1 — 强烈建议补充，否则测试与验收困难

#### 3.4 冲突检测规则与数据模型需对齐

§7.2 提到「同一节点 ID」「时间窗口 < 5 秒」「版本跳跃阈值」，但：

- 若采用 P0 中的逻辑 ID 方案，「同一节点 ID」的表述需改为「同一逻辑节点 / 同一 stable key」。
- 「版本跳跃」阈值未给出具体数值与触发后的行为（拒绝写入、强制拉取、标记冲突等）。

**PRD 涉及段落:** §7.2、§7.3、§5.2 `conflicts`。

#### 3.5 回滚的语义与多端一致性

`POST /api/v1/bookmarks/rollback` 仅描述 `version`。未说明：

- 回滚是否产生**新版本**（推荐：是，便于审计），还是原地覆盖；
- 已连接的插件如何获知需拉取（推送、轮询间隔、下次同步冲突）；
- 回滚期间并发写入的处置。

**PRD 涉及段落:** §5.4、§4.1.2、§4.1.4。

#### 3.6 设备注册与 API Key 的绑定流程

`POST /api/v1/devices` 的请求体含 `name`、`type`，未说明：

- 与 `X-API-Key` 的授权关系（单 Key 多设备是否允许）；
- `device_id` 是服务端签发还是插件自报；若自报，如何防伪造或重复。

**PRD 涉及段落:** §4.1.1、§5.2 `device_id`、§5.3。

#### 3.7 插件侧「监听书签变化」的实现级说明偏薄

PRD 写明使用 `chrome.bookmarks` API。扩展平台实际可提供 `onCreated`、`onRemoved`、`onChanged`、`onMoved` 等事件（具体以 Manifest V3 与权限为准）。

**建议:** 在 PRD 或附录中明确 **事件驱动为主、定时对账为辅**（防止漏事件），并约定对账周期；同时说明 `storage.sync` 配额对队列持久化的限制（见下）。

**PRD 涉及段落:** §4.2.2、§4.2.4、§8.2。

### P2 — 改进后文档更专业、可维护

#### 3.8 安全条款操作性不足

- §8.1「敏感操作需要二次验证」：未定义哪些操作、二次验证的形态（TOTP、懒猫账号密码、管理端确认）。
- §8.2「书签数据加密存储（可选）」：缺少密钥来源、轮换、备份恢复责任方。
- §8.3「操作审计日志」：未列字段（谁、何时、哪条 API、结果、来源 IP/设备）。

**PRD 涉及段落:** §8 全文。

#### 3.9 性能与体验验收需可测量定义

- 「1000+ 书签同步 < 10 秒」：未说明网络条件、是否冷启动、是否含图片/无 URL 文件夹等边界。
- 「内存占用 < 50MB」：指插件 Service Worker、后台页，还是懒猫进程，需指明。
- 「配置流程 < 3 分钟」：建议附**标准测试任务**（从安装到首次双向同步成功）。

**PRD 涉及段落:** §11.2、§11.3。

#### 3.10 风险表中的「跨域」表述需针对扩展场景校准

Chrome 扩展对 `host_permissions` 声明的懒猫 origin 发起请求时，通常**不受网页 CORS 同一套限制**。真正需写的是：**自签名证书 / 私有 CA、混合内容、企业策略拦截** 等。

**PRD 涉及段落:** §10 表格最后一行。

#### 3.11 非功能需求缺失

建议增加简短章节：**可用性（SLA 自用即可）**、**备份与恢复（DB 文件）**、**日志与排障**、**升级与迁移（DB schema）**、**国际化（若仅中文可写明 v1 仅 zh）**。

## 4. API 与设计一致性核对

| 项目 | 状态 | 说明 |
|------|------|------|
| `verify` 与后续请求 | 待澄清 | 未说明 Token/Session 还是仅 Key；若仅 Key，verify 可选 |
| `GET /bookmarks` 与 `since` | 待澄清 | 与 `version` 体系需统一（见 3.2） |
| `POST /sync` 中 `bookmarks` 与 `changes` 同时出现 | 待澄清 | 需明确是否冗余、何种情况下必填 |
| `conflict/resolve` 的 `merge` | 待澄清 | 未定义 merge 的字段级规则与响应形态 |
| 历史 `limit=50` 分页 | 待澄清 | 无 cursor，大数据量时需补充 |
| 错误响应格式 | 缺失 | 建议统一 `{ error_code, message, details }` 与 HTTP 状态码表 |

## 5. 开发计划与范围

- **14 天**完成核心服务 + 插件 + 增量 + 冲突 UI + 历史 + 多设备测试，在**未解决 P0（ID 模型）** 的前提下风险偏高；若 P0 需方案论证，建议单独列出「技术验证」子阶段或拉长 Phase 3。
- Phase 未包含 **最小安全基线**（Key 存储、HTTPS自签场景、速率限制），可在 Phase 1/4 中各加一条。

## 7. 结语

当前 PRD 已具备启动讨论的完整度；**唯一可能拖累整个 v1 的问题是跨设备书签标识与增量模型未定型**。建议在编写第一行 Go/React 代码之前，用简短设计说明（可附在 PRD 附录）冻结上述两点，其余条目可在迭代中增强。本文档审计结论以 `PRD-v1.md` 现文为准；若 PRD 更新版本号，可同步更新本报告对应关系。
