## ⚠️ 新发现的问题

### P1 - 开发中需注意

#### 1. Chrome 三根映射中"root"节点的 title 语言问题
**位置：** §4.5 + §13.4

映射表中 title 列有中文（"书签栏"）和英文（"Bookmarks Bar"）。但 Chrome 的根节点 title 取决于浏览器语言设置。中文 Chrome 显示"书签栏"，英文 Chrome 显示"Bookmarks Bar"。

如果插件在英文 Chrome 上运行，`chrome.bookmarks.getTree()` 返回的 title 是"Bookmarks Bar"，但服务端规范树中存的是"书签栏"。

**影响：** 首次上传时，插件按 Chrome 原生 title 传给服务端。服务端的固定 guid（root_bar 等）是硬编码的，但 title 应该以哪个为准？

**建议：**
- 服务端规范树第一层（root_bar、root_other、root_mobile）的 title 以客户端上报为准
- 或者：服务端不存储这三个节点的 title（它们是虚拟节点，仅用于结构）
- 新设备拉取时，第一层直接映射到 Chrome 固定根，不需要用 title 匹配

当前方案通过固定 guid 映射（root_bar → id="1"）而不是 title 匹配，所以实际上 title 不影响功能。建议在文档中明确"第一层节点的 title 仅供参考，映射基于固定 guid"。

#### 2. 同批 create 中 parent_local_id 与 parent_guid 的优先级
**位置：** §6.3.2

当变更同时提供了 `parent_guid` 和 `parent_local_id` 时，服务端应该用哪个？

示例：
```json
{
  "type": "create",
  "local_id": "101",
  "parent_local_id": "100",
  "data": { "parent_guid": "root_bar" }
}
```

两个都有值。逻辑上应该优先用 `parent_guid`（因为它是服务端已知 GUID），fallback 到 `parent_local_id`（同批创建的新节点）。

**建议：** 在文档中明确优先级：
```
如果 data.parent_guid 存在且非空 → 使用 parent_guid
否则如果 parent_local_id 存在 → 从同批缓存查找对应 GUID
否则 → 报错 E005
```

#### 3. 分页续页的 since_version 语义边界
**位置：** §6.3.1

分页续页方式是"使用最后一条变更的 to_version 作为下一次的 since_version"。但增量拉取谓词是 `to_version > since_version`，所以用最后一条的 to_version 作为下一次的 since_version，恰好不会重复拉取。

这是正确的，但如果两个变更的 to_version 相同（理论上不应该，因为每条变更 to_version 唯一），会有问题。鉴于版本号语义已经明确"每条变更 +1"，这个问题实际上不存在。但建议文档中补充一句"每条变更的 to_version 全局唯一"。

#### 4. 变更日志截断的"最近 30 天快照未引用"条件实现复杂度
**位置：** §5.1.2

物理删除条件是"软删除 90 天 AND 最近 30 天快照未引用"。检查"最近 30 天快照是否引用某个 GUID"需要扫描快照内容。如果快照是 gzip 压缩存储的，每次物理删除前解压扫描 500 个快照会有性能问题。

**建议：**
- 方案 A：维护一个引用计数表（guid → snapshot_count），每次创建快照时递增引用
- 方案 B：简化为"软删除 90 天后直接物理删除，不管快照引用"（快照本身 30 天后会清理）
- 方案 C：物理删除时只检查最近 10 个快照（而不是全部 500 个）

方案 B 最简单。考虑到快照 30 天清理 + 软删除 90 天清理，实际上被物理删除的书签不会出现在任何活跃快照中（除非有人手动查看历史版本）。v1 建议用方案 B，后续有需求再加引用检查。

### P2 - 实现细节

#### 5. 新设备拉取时，如果本地已有书签怎么办？
**位置：** §8.3

流程假设新设备的 Chrome 书签栏是空的。但如果用户在新设备上已经手动创建了一些书签，全量拉取时会与服务端数据产生冲突（本地已有书签 vs 服务端要创建的书签）。

**建议：** 补充新设备首次同步的前置检查：
- 如果本地已有书签且服务端也有数据 → 先执行全量上传（备份本地书签），再合并
- 或者：提示用户选择"合并"或"覆盖"

v1 可以简化为：新设备首次同步时，如果本地有书签，先自动全量上传（作为备份），然后拉取服务端数据做合并。

#### 6. 性能验收表中格式错误
**位置：** §12.2

表格中间出现了一个空的 `|` 行：
```
| 全量重拉（5000 书签） | < 30 秒 | 本地网络 |
```
这行没有与上面的表头对齐（缺少了一列"测试条件"的分隔）。不影响理解，但建议修复。

#### 7. 队列合并"仅合并未提交过的变更"的判断逻辑
**位置：** §5.2.4

"幂等保护：仅合并从未提交过的变更（无 client_request_id 记录）"——但 PendingChange 模型中没有 client_request_id 字段。变更是在推送时才被分配 client_request_id（整个 sync 请求级别），单条变更没有独立的提交记录。

**建议：** 改为"仅合并非 pending 状态的变更"或"通过 change_id 查找是否有已提交记录"。或者更简单地：队列合并时只合并同一 GUID 的连续变更，不区分是否已提交（因为已提交的变更在 applied_changes 返回后已经从队列移除）。

#### 8. E003 与 E008 的使用场景区分
**位置：** §6.1

E003（409）用于"base_version 严重落后需全量重拉"，E008（410）用于"变更日志溢出需全量同步"。两者都触发全量重拉，客户端行为相同。

**建议：** 客户端可以统一处理（收到 409 或 410 都走全量重拉），但建议在文档中补充两者的区别说明，避免开发者困惑。
