# PRD v2.0 审计报告

**审计人：** 大网管 (GLM-5) 🥷  
**审计日期：** 2026-04-16  
**文档版本：** PRD v2.0  
**审计范围：** 架构设计、API 规范、数据模型、同步逻辑、安全性、开发计划

---

## 总评

PRD v2.0 整体质量**中上**，作为个人/小团队项目的需求文档已经相当完整。v1→v2 的改动方向正确（删除 WebSocket 简化架构、明确增量同步模型、补充删除冲突处理）。但在同步模型的边界条件、API 设计一致性、数据模型冗余等方面存在需要进一步澄清的问题。

**综合评分：7.5 / 10**

---

## ✅ 做得好的地方

### 1. 双层 ID 映射设计合理
Chrome 不同设备本地 ID 不一致是书签同步的经典难题，用服务端 GUID + 本地映射表解决是业界标准做法，清晰且可落地。

### 2. 增量同步模型选择恰当
版本号 + Change Log 比全量 diff 轻量得多，1000 条变更日志的保留上限也合理。`client_request_id` 幂等设计体现了工程经验。

### 3. 冲突处理策略务实
四种冲突类型的分类清晰，"删除 vs 修改"默认保留修改（防误删）是好决策。冲突解决不是错误（HTTP 200）的设计也避免了客户端误判。

### 4. 回滚设计可审计
回滚生成新版本而非覆盖，保留了完整操作审计链，方便再次回滚。

### 5. 删除 WebSocket 的决策正确
对书签同步这种低频操作场景，轮询 30 秒完全够用，WebSocket 带来的复杂度远超收益。

## ⚠️ 需要关注的问题

### P0 - 架构级风险

#### 1. 变更日志 1000 条上限未定义溢出策略
**位置：** §4.2 "保留最近 1000 条"

当 change_log 超过 1000 条后怎么办？
- 直接截断旧记录？那新设备全量同步时拿不到完整历史。
- 如果新设备首次同步走的是全量拉取（GET /bookmarks），则 change_log 溢出只影响增量同步，影响有限。但文档没有明确说明这一点。

**建议：** 明确 change_log 的溢出行为。如果新设备首次同步走全量接口，则在文档中写清楚"change_log 仅用于增量同步，首次同步走全量接口"。另外建议 1000 条上限可配置。

#### 2. 同步流程图逻辑有分支缺失
**位置：** §8.1 完整同步流程图

流程图中"有本地变更？→否"和"有本地变更？→是"两条路径最终都汇聚到"同步完成"，但中间的 GET 和 POST 是并行还是串行没有说明。实际实现时应该是：

```
1. 先 POST 本地变更
2. 再 GET 远程变更（因为 POST 后版本号变了）
```

还是：

```
1. 先 GET 远程变更
2. 合并后 POST 本地变更
```

**建议：** 明确先推后拉还是先拉后推，并在流程图中体现顺序。推荐**先推后拉**（先提交本地变更，再拉取远程最新状态），避免拉取后到提交前产生新的竞争窗口。

#### 3. `expected_version` 乐观锁与冲突检测的交互不清晰
**位置：** §6.3.2 提交同步

请求体同时有 `base_version` 和 `expected_version`，但两者的区别和交互逻辑模糊：
- 如果 `expected_version` 不匹配就直接拒绝（返回 E003）？
- 还是仍然检测冲突并返回 conflicts 数组？
- 如果客户端发来的 `expected_version` 和 `base_version` 相同，这两个字段有什么区别？

**建议：** 简化为只用一个版本号字段。要么用乐观锁（`expected_version` 不匹配就拒绝），要么用冲突检测（不匹配就返回冲突列表让用户选）。同时用两种模式会增加客户端实现复杂度。

### P1 - 设计级问题

#### 4. API 设计风格不统一
**位置：** §6 全部 API

- 冲突解决用 `/conflict/resolve`（动词），设备撤销用 `DELETE /devices/<id>`（RESTful），回滚用 `/rollback`（动词）。
- 有的接口成功返回 `success: true` + `data`，有的直接返回字段（如设备列表返回 `devices` 而非 `data: { devices: [...] }`）。

**建议：** 统一为 RESTful 风格 + 一致的响应包装。例如：
- `POST /api/v1/bookmarks/sync/conflicts/{id}/resolve`（资源化）
- 所有响应统一为 `{ success, data, version, synced_at }` 结构

#### 5. `index` 字段的跨设备排序问题
**位置：** §7.1 Bookmark 数据模型

`index` 表示书签在父文件夹中的排序位置，但不同设备可能对同一文件夹下的书签有不同的排序偏好。如果两个设备同时移动同一文件夹下的不同书签，index 值会冲突。

**建议：** 
- 方案 A：`index` 由服务端统一分配，客户端不提交 index，服务端按 GUID 顺序或插入时间排列。
- 方案 B：用排序权重（如 float weight）代替绝对 index，避免冲突。

#### 6. 离线队列 5MB 限制的风险评估不足
**位置：** §5.2.4 后台服务

文档提到"降级策略：storage.local 满时暂停队列，提示用户清理"。但 5MB 对于书签数据来说其实不大——假设每条变更约 500 字节，5MB 约能存 10000 条变更。如果用户长期离线（比如出差一周），累积的变更可能超限。

**建议：** 
- 增加队列条数上限（如 5000 条），超限后合并变更（同一 GUID 的多次变更合并为一条）。
- 补充变更合并策略说明。

#### 7. 软删除 30 天物理删除策略可能太激进
**位置：** §5.1.2 书签存储

30 天后物理删除意味着无法恢复 30 天前删除的书签（即使历史版本保留 50 个）。如果一个用户很少同步（比如两台设备，一台关了一个月），可能错过恢复窗口。

**建议：** 
- 将物理删除周期延长到 90 天，或者让物理删除与历史版本联动（有历史版本引用的已删除书签不物理删除）。
- 也可以提供用户配置入口。

#### 8. 历史版本 "保留最近 50 个" 的定义不清
**位置：** §5.1.2 书签存储

"版本"是什么粒度？
- 每次成功 sync 就是一个版本？那高频同步用户（30 秒一次）一天就有 2880 个版本，50 个只能覆盖约 25 分钟。
- 还是每次有实际变更才算一个版本？

**建议：** 明确版本的定义。如果是每次 sync 就是一个版本，50 个远远不够。建议按时间保留（如最近 7 天）或提高到 500+。

### P2 - 实现细节

#### 9. `date_added` / `date_modified` 使用客户端时间
**位置：** §7.1 Bookmark 数据模型

`date_added` 和 `date_modified` 是 Unix 毫秒时间戳，但没有说明是客户端时间还是服务端时间。§4.3 提到"冲突检测使用服务端接收时间"，但数据模型中的时间字段没有明确。

**建议：** 所有时间字段统一使用服务端时间（以服务端时钟为准），客户端时间仅作为参考字段（`client_date_added`）。

#### 10. 认证模型过于简单
**位置：** §5.1.1 / §6.2

当前只有 API Key 认证，没有用户概念。如果未来要支持"团队用户"（§1.4 提到的 v2 规划），API Key 模型无法扩展到多用户。

**建议：** v1 可以保持简单，但建议在 API 路径中预留用户空间（如 `/api/v1/users/me/bookmarks`），避免 v2 需要改 API 路径。

#### 11. 缺少书签数量上限说明
文档没有说明服务端对书签总数或单次同步变更数的限制。

**建议：** 明确：
- 单个用户书签上限（建议 10000 或更高）
- 单次 sync 请求的 changes 数组上限（建议 500）
- 单个文件夹的子节点上限

#### 12. 缺少错误恢复场景
**位置：** §8 同步逻辑

同步中途失败（如网络断开、服务端 500）后的恢复策略不清晰。例如：
- POST /sync 返回 500，客户端是否应该重试？重试时 `client_request_id` 相同是否会幂等？
- 部分变更已应用但响应丢失，客户端怎么知道哪些已生效？

**建议：** 补充错误恢复场景的流程说明，特别是 `client_request_id` 幂等机制的具体行为（是返回上次结果还是返回当前状态？）。

#### 13. 插件 Service Worker 休眠问题低估
**位置：** §11 风险

Manifest V3 的 Service Worker 会在 5 分钟不活跃后休眠。文档提到用 `chrome.alarms` 解决，但 `chrome.alarms` 最小间隔是 1 分钟（发布版本），而且休眠期间 `chrome.bookmarks` 事件监听会丢失。

**建议：** 
- 明确接受"休眠期间的书签变更在下次唤醒后通过全量 diff 检测"的降级策略。
- 或使用 `chrome.bookmarks.onChanged` 等事件通过 `chrome.events` 保持 Service Worker 活跃（但这有被 Chrome 限制的风险）。
- 建议在插件启动和 alarms 触发时，对比当前书签树和上次同步快照来检测遗漏变更。

## 总结

PRD v2.0 是一份质量不错的项目需求文档，核心设计（双层 ID、增量同步、冲突处理）方向正确且可落地。主要问题集中在**边界条件定义不清**和**API 风格不统一**。建议在开发前解决 P0 级别的 3 个问题，P1 级别问题可以在开发过程中逐步澄清。

**推荐开发优先级调整：** Phase 1 先做核心 API（不含冲突解决），Phase 3 再补充冲突检测。先用"最新优先"的简单策略跑通全流程，复杂冲突解决可以迭代。
