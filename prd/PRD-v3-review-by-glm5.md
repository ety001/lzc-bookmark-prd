## ⚠️ 需要关注的新问题

### P0 - 必须在开发前解决

#### 1. POST /sync 的冲突语义与"先推后拉"存在矛盾
**位置：** §6.3.2 + §8.1

"先推后拉"的流程是：POST 本地变更 → 有冲突则暂停 → 无冲突则 GET 拉取。但冲突检测发生在 POST 阶段，意味着**部分变更已经被服务端应用**（`applied_changes` 不为空），而冲突的变更被挂起。

问题：被挂起的冲突变更是否会阻塞后续所有同步？如果用户一直不解决冲突，其他无冲突的本地变更怎么办？

**建议：**
- 明确冲突是**阻塞式**还是**非阻塞式**。推荐非阻塞：无冲突的变更正常应用，只有冲突的变更挂起等待用户解决。当前设计看起来已经是这样（`applied_changes` 和 `conflicts` 可以同时返回），但需要在文档中明确写出来。
- 补充：如果客户端有新的本地变更，但还有未解决的冲突，是否可以继续推送非冲突变更？

#### 2. 新设备全量拉取后，如何创建 Chrome 书签？
**位置：** §8.3

流程说"插件为每个节点生成/分配 local_id"，但 Chrome API 不支持手动指定 ID。实际流程应该是：

```
1. 拉取服务端规范树
2. 调用 chrome.bookmarks.create() 逐个创建书签（Chrome 自动分配 local_id）
3. 建立 local_id ↔ guid 映射表
```

但这里有排序依赖问题：创建子书签前必须先创建父文件夹（因为 Chrome 要求 parentId 存在）。如果规范树很深（比如 5 层嵌套），需要 BFS/DFS 逐层创建。

**建议：** 补充新设备全量拉取的详细步骤，特别是树的创建顺序（BFS 从根到叶）和文件夹与书签的创建顺序。

#### 3. index_weight 归一化的触发时机和影响范围
**位置：** §4.4

归一化（"权重空间耗尽时重新归一化所有权重为 1.0, 2.0, 3.0..."）会**改变同一文件夹下所有书签的 index_weight**，这意味着：
- 归一化后，这些书签的 index_weight 变了，会生成大量变更记录
- 其他设备的增量同步会收到一堆 move 类型的变更
- 如果频繁触发归一化，会产生变更日志膨胀

float64 的精度约为 15-16 位有效数字。假设从 1.0 开始，每次在两个相邻节点中间插入，大约连续二分 50 次后精度就会耗尽。50 次在同两个节点间插入在实际使用中不太常见，但不是不可能。

**建议：**
- 明确归一化是一个低频操作（可设阈值，如精度差 < 1e-10 时触发）
- 归一化生成的变更是否计入变更日志？建议不计入（或作为特殊的 "reindex" 类型，客户端可以忽略）
- 或者考虑用整数间隔方案（如 [0, 1000000]），只在极端情况下归一化

### P1 - 开发中需要澄清

#### 4. client_request_id 的幂等行为未精确说明
**位置：** §8.6

"重试时服务端检测 client_request_id，返回上次结果"——但"上次结果"是什么？
- 是返回上次生成的 new_version 和 applied_changes？
- 还是返回当前最新状态（可能有其他设备的变更）？

如果返回上次结果，那客户端可能错过上次请求后其他设备提交的变更。

**建议：** 幂等行为应该是"检测到重复的 client_request_id，不重复应用变更，但返回当前的冲突状态和最新版本号"。即幂等的是"不重复应用变更"，响应仍然反映当前最新状态。

#### 5. 限流维度 "API Key + Device" 的实现复杂度
**位置：** §6.6

同一 API Key 下不同设备独立限流，需要在限流器中维护 `api_key + device_id` 组合的计数器。如果 API Key 可以绑定"不限设备"，那限流器的 key 数量可能很大。

**建议：** 
- 增加"单 API Key 总限流"（如 200 次/分钟）作为上限兜底，防止一个 API Key 下的所有设备总量超限
- 或者简化为"单设备限流"（只看 device_id），降低实现复杂度

#### 6. 全量 diff 检测（Service Worker 降级）的实现可行性
**位置：** §5.2.4

"对比当前树与上次同步快照"听起来简单，但：
- `chrome.bookmarks.getTree()` 返回的树可能包含数千个节点
- 在 storage.local 中保存完整快照会占用大量空间（与 5MB 限额竞争）
- diff 算法的实现和调试都不简单

**建议：**
- 评估快照大小：5000 个书签的快照约 500KB-1MB，可以存在 storage.local 但要谨慎
- 建议只保存关键元数据的摘要（guid + title + url + parent_guid 的 hash），而不是完整书签数据
- 或者干脆简化为：Service Worker 唤醒时，总是先做一次 GET /bookmarks?since_version=N 拉取服务端变更，然后对比本地变更推送。不做本地 diff。

#### 7. 创建冲突的 URL 去重逻辑可能过于激进
**位置：** §4.3

"两个设备创建同名同 URL 书签（无 GUID）→ URL 去重，合并为同一 GUID"

用户可能在两个设备上故意创建同名同 URL 的不同书签（放在不同文件夹下）。自动合并可能不符合预期。

**建议：** 创建冲突不应该自动合并。应该：
- 两个书签都保留，各自分配不同的 GUID
- 或者提示用户选择（保留一个 / 保留两个）

#### 8. 变更日志保留条件可能导致不一致
**位置：** §4.2

保留条件是 `COUNT <= 1000 AND age <= 7 days`，意味着：
- 如果 7 天内变更超过 1000 条，会截断到 1000 条
- 如果 7 天后变更不足 1000 条，仍会被清理

但 `from_version >= last_known_version` 的过滤条件依赖变更日志的连续性。如果截断了中间的变更，客户端的 `since_version` 可能落在被截断的区间内。

**建议：**
- 截断时必须更新基准快照（`is_baseline=true` 的 snapshot.version 更新为截断点的版本号）
- 客户端拉取时，如果 `since_version < baseline_version`，直接返回 410 GONE（v3 已经这样设计了，但需要在变更日志截断逻辑中同步更新 baseline）

### P2 - 实现细节

#### 9. API 认证接口返回 `version: 0` 语义不清
**位置：** §6.2

`POST /auth/verify` 返回 `"version": 0`。但 version 是书签同步的版本号，与认证接口无关。所有非书签接口都返回 `version: 0` 吗？

**建议：** 非书签接口的 `version` 字段设为可选，或返回 `null`。硬编码 0 可能被误解为"书签版本 0"。

#### 10. 500 个版本快照的存储成本未评估
**位置：** §5.1.2

500 个版本快照 × 每个快照包含完整书签树。假设 5000 个书签，每个快照约 500KB，500 个快照 = 250MB。对于 SQLite 来说不算多，但在懒猫微服的存储限制下需要评估。

**建议：** 补充快照的存储估算和压缩策略（如 gzip 压缩后约 50-100MB）。

#### 11. 回滚操作的原子性
**位置：** §6.5

`POST /bookmarks/rollback` 将书签树回滚到历史版本。但如果回滚过程中有其他设备正在推送变更怎么办？

**建议：** 回滚操作应该：
- 加全局写锁（或基于版本号的乐观锁）
- 回滚完成后生成新版本
- 其他设备的并发推送应该等待或收到冲突

#### 12. 插件本地备份功能可能造成困惑
**位置：** §5.2.1

"每次同步前自动导出书签 JSON 到 chrome.downloads"——这意味着用户下载目录会不断出现书签备份文件。

**建议：**
- 默认关闭此功能
- 或者用 IndexedDB 替代 chrome.downloads 存储备份
- 如果一定要用 chrome.downloads，限制保留最近的 N 个备份文件

#### 13. 并发验收标准缺少性能指标
**位置：** §12.5

验收标准只有"无数据丢失"和"冲突检测准确"，没有并发场景下的性能指标。

**建议：** 补充：
- 3 设备并发同步时，单次同步响应时间 < 5 秒
- 10 设备同时全量拉取，P95 响应时间 < 60 秒

## 总结

PRD v3.0 是一份**接近可开发**的需求文档。v2 审计中的主要问题（溢出策略、同步顺序、版本号机制、API 格式、index 冲突）都已得到妥善解决。剩余的 P0 问题（冲突语义、新设备拉取步骤、index_weight 归一化）可以在开发启动会上快速对齐。

**建议下一步：**
1. 开发启动会过一遍 P0 的 3 个问题（预计 30 分钟）
2. Phase 1 可以直接启动，不用等全部问题解决
3. 建议先跑通"单设备全量备份→恢复"的最小闭环，再叠加增量同步和冲突处理
