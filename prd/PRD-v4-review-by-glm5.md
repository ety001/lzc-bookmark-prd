## ⚠️ 新发现的问题

### P1 - 开发中需注意

#### 1. 变更日志溢出保留条件存在逻辑歧义
**位置：** §4.2

v3 写的是 `COUNT <= 1000 AND age <= 7 days`（满足两个条件才保留），v4 改为 `COUNT <= 1000 OR age <= 7 days`（满足任一即保留）。

v4 的 OR 语义意味着：只要变更不超过 7 天，即使超过 1000 条也保留。这可能导致变更日志无限增长（高频同步用户 7 天内可能产生上万条变更）。

**建议：** 改为更精确的表述：
- 变更日志保留 `min(1000 条, 7 天内)` —— 取两个条件的交集
- 或改为 `COUNT <= 1000 AND age <= 30 天`（任一不满足则可截断）

当前 OR 语义在实际实现中可能导致内存/存储压力，需要确认这是有意为之还是笔误。

#### 2. 部分冲突后"下次推送仅推送冲突变更 + 新增变更"的边界情况
**位置：** §6.3.2

客户端更新了 `last_known_version = new_version`（如 128），但冲突变更的 `base_version` 仍然是旧的（如 120）。下次推送冲突变更时，`base_version` 应该用什么？

- 如果用 120（原始 base_version）：服务端可能认为冲突已经不存在了（因为版本已经推进到 128）
- 如果用 128（新的 last_known_version）：那冲突变更的检测逻辑需要重新评估

**建议：** 明确冲突变更重新推送时的 `base_version` 应该设为 `new_version`（冲突解决时的最新版本号），而不是原始的 `base_version`。服务端应该根据冲突记录的 `conflict_id` 来匹配，而不是依赖 `base_version`。

#### 3. `reindex` 变更不计入上限但需要计入变更日志
**位置：** §4.4 + §7.3

"reindex 不计入变更日志上限"——但 reindex 变更仍然写入 `change_log` 表（否则客户端无法感知排序变化）。不计入上限意味着它不会触发溢出截断，但在存储层面它仍然占用空间。

**建议：** 明确 reindex 变更的存储策略：
- 写入 change_log，但不计入 1000 条溢出判断
- 或者单独存储 reindex 记录，客户端增量拉取时合并返回

前者更简单，后者更清晰。对于 v1 建议用前者。

#### 4. 回滚生成的 `rollback` 变更的 `data` 字段内容未定义
**位置：** §7.3

Change 类型的 `data` 字段对 `rollback` 类型应该包含什么？完整树差异？还是仅包含版本号引用？

**建议：** `rollback` 变更的 `data` 应包含：
```json
{
  "rollback_to_version": 100,
  "affected_guids": ["guid1", "guid2", ...]  // 受影响的 GUID 列表
}
```
客户端收到 `rollback` 变更时，不需要从 `data` 重建树（因为可以拉取快照），但需要知道哪些 GUID 受影响以更新本地映射。

### P2 - 实现细节

#### 5. 全量上传分批提交的 batch 边界
**位置：** §8.2

首次同步"可分批提交（每批 500 条变更）"，但分批提交时：
- 第一批的 `base_version=0`，后续批的 `base_version` 是上一批返回的 `new_version`
- 每批都有独立的 `client_request_id`
- 如果第三批失败了，前两批已应用的变更怎么回滚？

**建议：** 首次全量上传建议用事务机制：
- 方案 A：服务端支持"会话"概念，开始上传 → 多批提交 → 确认完成。未确认的会话超时回滚。
- 方案 B：简单处理——如果某批失败，重试该批；如果最终部分成功部分失败，客户端基于当前状态继续同步即可（因为书签是覆盖语义，不会丢失）。

方案 B 对 v1 更实用，建议在文档中明确选择方案 B。

#### 6. BFS 创建时 Chrome 根节点的映射问题
**位置：** §8.3

Chrome 有 3 个固定根节点：`0`（根）、`1`（书签栏）、`2`（其他书签）。服务端的规范树只有一个 `root`。

新设备全量拉取时，服务端的 `root` 应该映射到 Chrome 的哪个根？是 `0`（根）还是 `1`（书签栏）？

**建议：** 明确映射规则：
- 服务端 `root` → Chrome 根节点 `0`
- 服务端规范树的第一层子节点分别映射到 Chrome 的 `1`（书签栏）和 `2`（其他书签）
- 或者：服务端规范树只管理"书签栏"（`1`），不管理"其他书签"和"移动设备书签"

建议后者更安全——只同步"书签栏"，避免干扰 Chrome 的其他书签区域。

#### 7. `has_more` 字段缺少分页机制
**位置：** §6.3.1（v3 遗留）

增量响应中有 `has_more: false`，但 GET 请求没有分页参数。如果变更日志在 7 天内积累了 5000 条（OR 语义下可能），一次性返回所有变更可能导致响应过大。

**建议：** 补充分页参数：
```
GET /api/v1/bookmarks?since_version=120&limit=100
```
如果 `has_more=true`，客户端用返回的最后一条 `to_version` 作为下一次请求的 `since_version`。

#### 8. 只读 API Key 的权限粒度
**位置：** §5.1.1

只读 API Key 禁止调用 POST /sync、POST /conflicts/resolve、POST /rollback。但只读 Key 是否可以调用：
- `POST /api/v1/devices`（注册设备）？
- `GET /api/v1/bookmarks`（拉取书签）？
- `GET /api/v1/bookmarks/history`（查看历史）？

根据语义，GET 请求应该允许，POST 设备注册也应该允许（因为它不改变书签数据）。

**建议：** 补充只读 API Key 的完整权限矩阵。
