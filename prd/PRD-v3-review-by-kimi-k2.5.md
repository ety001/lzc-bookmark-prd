# 懒猫书签同步系统 PRD v3.0 - 审计报告

**审计者:** kimi-k2.5  
**审计时间:** 2026-04-16  
**PRD 版本:** v3.0  

---

## 总体评价

PRD v3.0 相比 v2.0 有**实质性改进**，核心数据模型更加简洁清晰。主要改进包括：
- ✅ 服务端移除 `local_id`，仅存储 GUID 规范树，职责分离更明确
- ✅ 版本号机制简化（`from_version`/`to_version` 替代 `base_version`/`expected_version`）
- ✅ 明确"先推后拉"同步顺序，避免竞争窗口
- ✅ 新增 410 GONE 溢出处理机制
- ✅ 延长软删除周期至 90 天
- ✅ 补充了大量边界条件和错误恢复场景

文档已达到**生产级 PRD** 标准，可直接指导开发。

## 详细审计意见

### 1. 产品概述 ✅

**改进:**

---

### 2. 用户场景 ✅

**新增场景:**
- "长期离线后同步"场景（变更日志溢出 → 全量重拉）

---

### 3. 系统架构 ✅

**重大改进:**
- **服务端仅存储 GUID 规范树，不存储 Chrome local_id**
- **local_id ↔ GUID 映射仅存在于各插件本地**

这是 v3 最核心的架构改进，职责分离清晰：
- 服务端：纯粹的 GUID 规范树管理
- 插件端：负责 local_id 与 GUID 的映射转换

---

### 4. 核心设计决策 ✅

#### 4.1 书签身份模型 ✅

**v3 重大改进：**
```typescript
// 服务端（规范树，不含 local_id）
Bookmark { guid, tenant_id, title, url, parent_guid, index_weight, ... }

// 插件本地（映射表，不上传）
IdMapping { local_id, guid, last_synced_version }
```

**优点:**
- 服务端数据模型简洁，无设备相关字段
- 插件完全负责 local_id 管理，服务端无感知
- 新设备首次拉取时，自行建立映射表

**建议:**
- 首次全量上传时，服务端返回 `generated_mappings` 的设计合理
- 考虑新设备首次拉取后，如何生成 `local_id`？（Chrome API 创建书签时返回）

#### 4.2 增量同步模型 ✅

**版本号机制改进：**
```typescript
// v2
{ base_version: 120, expected_version: 122 }

// v3
Change { from_version: 120, to_version: 121 }
```

**优点:**
- `from_version`/`to_version` 语义更清晰
- 每个变更独立记录版本范围，便于追溯

**变更日志溢出策略 ✅**
- 保留条件：`COUNT <= 1000` AND `age <= 7 days`
- 溢出后返回 410 GONE，触发全量重拉
- 基准快照（`is_baseline=true`）设计合理

**建议:**
- 明确 7 天和 1000 条的优先级（AND 还是 OR？）
- 建议：超过 7 天**或**超过 1000 条即触发溢出

#### 4.3 冲突检测与解决 ✅

**新增冲突类型:**
- "创建冲突"：两个设备创建同名同 URL 书签（URL 去重合并）

**解决流程清晰:**
- 先推后拉顺序避免竞争窗口
- 冲突解决后再拉取最新状态

#### 4.4 排序权重设计 ✅

**优秀设计:**
- 使用 `index_weight: float` 代替绝对 `index`
- 避免多设备同时移动书签导致的 index 冲突
- 权重空间耗尽时重新归一化

**建议:**
- 明确 float 精度（建议 double / float64）
- 归一化触发条件：最小间隔 < 0.001？

### 5. 功能需求 ✅

**改进:**
- 软删除周期：30 天 → 90 天
- 版本历史保留：50 个 → 500 个 / 30 天
- 离线队列上限：明确 5000 条
- 队列超限策略：合并同一 GUID 的连续变更

**新增:**
- 本地备份：导出 JSON 到 chrome.downloads（可选，默认开启）
- Service Worker 休眠降级：全量 diff 检测

---

### 6. API 设计 ✅

#### 6.1 通用规范 ✅

**新增:**
- `X-Request-Id` 可选，用于全链路追踪

#### 6.2 书签接口 ✅

**GET /api/v1/bookmarks** ✅

**改进:**
- 全量响应包含 `is_baseline` 和 `baseline_version`
- 增量响应包含 `has_more`（分页指示）

**新增 410 GONE 响应:**
```http
HTTP/1.1 410 GONE
{
  "error": {
    "code": "E008",
    "message": "变更日志已溢出，请全量同步",
    "details": { "baseline_version": 100, "current_version": 500 }
  }
}
```

设计合理，客户端可据此触发全量重拉。

**POST /api/v1/bookmarks/sync** ✅

**改进:**
- `changes` 数组上限 500 条/请求
- `generated_mappings` 仅返回 create 类型的 GUID 分配
- `applied_changes` 返回具体变更 ID 列表

**响应设计:**
- 成功：返回 `new_version`, `applied_changes`, `conflicts`
- 有冲突：HTTP 200， conflicts 数组非空

#### 6.3 冲突解决接口 ✅

**路径改进:**
```
POST /api/v1/bookmarks/conflicts/{conflict_id}/resolve
```

比 v2 的 `POST /api/v1/bookmarks/conflict/resolve` 更 RESTful。

**merge 策略:**
- `parent_guid`/`index_weight` 必须与 local 或 remote 一致
- 仅 `title`/`url` 可合并

#### 6.4 数据上限 ✅

**新增章节:**
| 限制项 | 上限 | 超限行为 |
|--------|------|----------|
| 单用户书签总数 | 100,000 | 返回 E005 |
| 单次 sync 变更数 | 500 | 分批提交 |
| 离线队列变更数 | 5000 | 合并同一 GUID 变更 |

### 7. 数据模型 ✅

**改进:**
- Bookmark 移除 `local_id`，新增 `tenant_id`（预留）
- Bookmark 新增 `index_weight`（float）
- Bookmark 新增 `deleted_at`（软删除时间）
- Change 使用 `from_version`/`to_version`
- 新增 Snapshot 模型（版本快照）

**建议:**
- Snapshot 的 `bookmark_tree` 存储方式（JSON 序列化？）
- 考虑快照存储空间优化（与变更日志重复存储？）

---

### 8. 同步逻辑 ✅

**重大改进：明确"先推后拉"顺序**

```
1. 检查本地变更 → 2. POST /sync（推送） → 3. GET /bookmarks（拉取）
```

**优点:**
- 避免拉取后到提交前的竞争窗口
- 推送后服务端版本已更新，拉取的是最新状态

**新增流程:**
- 首次同步流程（全量上传）
- 新设备首次同步（全量拉取）
- 错误恢复场景（500、超时、410、网络不可用）

**错误恢复场景表 ✅**
- 500 错误：不更新版本号，下次重试
- 超时：指数退避重试 3 次
- 410 GONE：清空映射表，全量重拉
- 网络不可用：离线队列持久化
