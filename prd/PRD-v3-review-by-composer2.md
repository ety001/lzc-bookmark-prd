## 1. 总体结论

v3 在 v2 基础上**系统性吸收**了此前评审中的要点：服务端 **移除 `local_id`**、**HTTPS/HTTP 例外写进约束**、**先推后拉**、**410 / E008 全量重拉**、**限流维度（API Key + Device）**、`merge` 的 **`merged_data` 示例**、**离线队列上限与合并策略**、**Service Worker 休眠时的全量 diff 降级**、**index冲突用 `index_weight`** 等，文档完整度与可执行性已达到**较高水平**。

当前最需要在实现前**钉死**的是：**全局版本号与单条 `Change` 的 `from_version`/`to_version` 是否一一对应**（一批 `sync` 会推进多少版本），以及 **增量拉取的过滤谓词**与之一致；否则客户端 `last_known_version`、幂等与冲突处理会对不齐。

**整体评级:** 优秀  
**建议状态:** 可进入详细设计与开发，**补充 §4.2/§6.3.1/§7.3 的版本语义一小节**后风险更低。

## 3. 关键问题与建议（按优先级）

### P0 — 建议在开发前写入 PRD（或《同步协议》附录）

#### 3.1 版本号粒度：`Change` 与「每次成功同步 +1」的关系

§4.2 `GlobalState` 写「每次成功同步 +1」，同时 `Change` 写 `to_version = from_version + 1`，暗示**每条变更**对应版本 +1。§6.3.2 又允许单次请求最多 **500 条** `changes`。

**缺口:** 若一批成功应用 50 条变更，全局 `current_version` 是 **+50** 还是 **+1**？`change_log` 是一条合成记录还是 50 条？这直接决定：

- `GET ...?since_version=` 的过滤条件应用 `to_version > N` 还是别种谓词；  
- 冲突「部分应用」后 `new_version` 与客户端 `last_known_version` 如何更新。

**建议:** 二选一写死并贯穿 §4.2、§6.3、§7.3、§8.1：

- **方案 A（细粒度）:** 每应用一条变更，全局版本 +1；一批50 条则 +50，`change_log` 存 50 条。  
- **方案 B（粗粒度）:** 每个成功 `sync` 请求全局 +1，日志内多条变更共享同一 `to_version` 或改用「批次 id」；则 `Change` 上的 `from_version`/`to_version` 语义需重写。

当前文档两种表述并存，属于**最高优先级**澄清项。

**涉及段落:** §4.2（GlobalState、Change、同步流程）、§6.3.1、§6.3.2、§7.3、§8.1。

#### 3.2 增量拉取过滤条件与 `last_known_version` 对齐

§4.2 写：返回 `change_log` 中 `from_version >= last_known_version` 的变更。更稳妥的常见定义是：客户端在版本 **N** 已与服务器状态对齐，应拉取所有满足 **`to_version > N`** 的条目（避免重复应用或边界差一）。

**建议:** 在确定3.1 的版本语义后，用**形式化一句**定义拉取谓词，并给一例：已知 `last_known_version=120`，日志中三条变更的版本区间，应返回哪几条。

**涉及段落:** §4.2、§6.3.1（增量响应）、§8.1。

#### 3.3 §4.2 步骤 1 的接口笔误

§4.2「同步流程」第 1 步写：携带 `last_known_version` 请求 **`/api/v1/bookmarks/sync`**。增量拉取在 v3 中应由 **`GET /api/v1/bookmarks?since_version=`** 完成（与 §3.4、§6.3.1 一致）。

**建议:** 将该步改为 GET，或将该段改为「推送前客户端已知版本来自上次 GET/推送响应」，避免读者照抄实现错误路径。

**涉及段落:** §4.2。

### P1 — 强烈建议补充，减少产品与联调争议

#### 3.4 「创建冲突」URL 去重合并

§4.3 写：两设备创建**同名同 URL** 无 GUID 时 **URL 去重合并为同一 GUID**。用户常有意在不同文件夹放同一 URL（或同一标题），合并会导致**非预期丢条目**。

**建议:** 改为「可选策略」或缩小为「同一 `parent_guid` 且 URL 相同」再合并，并在验收中增加**负例**（同 URL 不同文件夹须保留两条）。

**涉及段落:** §4.3。

#### 3.5 首次全量上传：树序与文件夹

§8.2 写：`changes` 包含所有节点 `type=create`。需约定 **父节点先于子节点**（或分多批拓扑序），否则服务端无法解析 `parent_guid`。

**建议:** 在 §8.2 或 §6.7 增加「全量 create 须拓扑有序」或「服务端按 guid 占位/多趟处理」的明确规则。

**涉及段落:** §8.2、§6.3.2。

#### 3.6 推送成功后拉取：`since_version` 取值

§8.1 流程写推送后 `GET ...?since_version=M`，M 为服务端更新后的版本。若 M 表示「当前全局头版本」，则增量接口应定义 **`since_version=M` 是否包含在「已包含状态」内**（通常应拉取 **`to_version > M`** 的空集，或改用 `since_version=推送前客户端所知版本`）。

**建议:** 与 3.1、3.2 一并写清：推送响应中的 `new_version` 与随后拉取的参数如何衔接，避免漏拉或重复拉。

**涉及段落:** §3.4、§8.1、§6.3.1。

#### 3.7 冲突且部分应用时客户端状态

§6.3.2「有冲突」示例中 `new_version` 仍为 123，部分 `applied_changes` 已生效。需说明：

- 未解决冲突前，客户端是否更新 `last_known_version`；  
- 本地队列是否剔除已确认应用的 `change_id`，冲突项是否保留待 resolve。

**建议:** 在 §8.4 或 §8.6 增加数条**确定性规则**。

**涉及段落:** §6.3.2、§8.1、§8.6。

#### 3.8 软删除 90 天物理删除 vs 快照/历史

§8.5 写物理删除条件含「无历史版本引用」。需约定：快照、回滚、审计是否阻止 GC，否则实现易分歧。

**建议:** 一句策略即可（例如：快照保留期内不物理删书签行，仅标记）。

**涉及段落:** §8.5、§5.1.2、§6.5。

#### 3.9 `index_weight` 浮点与精度

§4.4 已考虑归一化，仍建议在实现提示中注明：**比较与排序使用稳定规则**（例如固定小数位或 rational），避免不同语言浮点显示顺序不一致。

**涉及段落:** §4.4、§7.1。

### P2 — 文档与体验 polish

#### 3.10 每次同步前导出 JSON 到 downloads

§5.2.1 默认开启可能带来**频繁下载文件、权限与打扰**。建议 PRD 注明：**默认关闭**或「每日首次同步 / 仅重大变更前」等，与 §11 风险表中「覆盖」缓解措施一致即可。

**涉及段落:** §5.2.1、§11。

#### 3.11 附录状态机 §13.2

「IDLE → SYNCING → PULLING」未显式区分**推送阶段**与拉取；与「先推后拉」可标为 PUSHING → PULLING 或在图中标注 sync 含 POST。

**涉及段落:** §13.2。

#### 3.12 术语表版本措辞

§13.1 等处「v1 暂不加密」「版本号」与产品 v3 文档版本并存，可读性略混。可改为「当前阶段」或「书签同步 v1 能力集」。

**涉及段落:** §9.2、§13.1。

#### 3.13 PostgreSQL 切换

§3.3有 `DB_TYPE=postgres`，§10 Phase 1 仍仅 SQLite schema。可接受；若与 v2 评审一致，可加**一句迁移/同构 schema** 指向（不必展开）。

**涉及段落:** §10、§12.4。

| 项 | 结论 |

| 服务端无 `local_id` | 一致 |
