## 1. 总体结论

PRD v5.0 在 v4.0 的基础上完成了**所有关键问题的收尾**。v4 评审中提出的 2 个 P0 问题（change_log 截断逻辑、Chrome 三根映射）和 7 个 P1/P2 问题（冲突时间窗口、全部冲突 version 语义、reindex 格式、离线队列幂等、分页参数、弱网基准、术语歧义）均已被系统性地吸收并写进文档。此外，v5 还新增了**同批 create 的 GUID 分配协议**、**409 E003 与 200 conflicts 的边界**、**只读 API Key 权限矩阵**、**分批提交策略**、**压力验收**等极具工程价值的细节。

当前文档的**完整性、一致性和可执行性均已达到生产级标准**。剩余问题均为实现层面的边界细节，不会阻塞开发启动。

**整体评级:** 优秀（生产就绪）  
**建议状态:** **可直接进入开发**，建议在技术设计阶段将 P1 问题（分页截断、reindex 计数）写入实现备忘。

## 3. 关键问题与建议（按优先级）

### P0 — 已解决，无遗留

v4 评审中的 2 个 P0 问题在 v5 中均已完美解决：

1. **change_log 截断逻辑**（§4.2）：OR 触发截断、AND 保留结果，逻辑自洽。
2. **Chrome 三根映射策略**（§4.5 / §13.4）：规范树第一层固定为 3 个系统根，新设备拉取时直接映射到 Chrome 固定根，不调用 `create()`，解决了 v4 中最大的实现盲区。

### P1 — 开发中建议关注，建议写入技术设计备忘

#### 3.1 分页拉取过程中变更日志截断的边界情况

**位置:** §6.3.1「分页行为」

场景：
```
1. 客户端请求 GET /bookmarks?since_version=100&limit=1000
   → 服务端返回 to_version 101-1100，has_more=true
2. 客户端继续请求 GET /bookmarks?since_version=1100&limit=1000
   → 在此期间变更日志被截断，baseline_version 更新为 1500
   → since_version=1100 < baseline_version=1500，服务端返回 410 GONE
```

**影响:** 客户端已经应用了 1000 条变更（version 1100），突然被迫全量重拉。虽然功能上不会丢失数据，但会导致额外的全量同步开销，且用户体验上可能出现"同步到一半又从头开始"的困惑。

**建议:** 在技术设计中约定一种策略：
- **方案 A（推荐）:** 服务端对分页请求使用**快照读**（或记录请求开始时的 baseline_version）。如果分页序列已开始，即使中间发生截断，也继续返回剩余变更，确保客户端能完整拉取到当前版本。
- **方案 B:** 服务端返回 410 时附带 `already_applied_version: 1100`，客户端知道已应用的部分，全量重拉后只应用 `to_version > 1100` 的部分（但全量响应不含 change_log，无法做到）。

**建议采用方案 A**，并在 §6.3.1 分页行为末尾增加一句：
> 「服务端应保证同一分页序列的连续性：若客户端基于上一页返回继续请求，即使期间触发截断，也应返回剩余变更或空结果，而非 410。」

**涉及段落:** §6.3.1、§4.2。

#### 3.2 `reindex` / `rollback` 变更是否参与 `COUNT(change_log)` 统计

**位置:** §4.2「变更日志截断策略」、§4.4「归一化策略」、§7.3「特殊变更类型」

文档明确 `reindex` 变更「不计入变更日志 1000 条上限判断」，但仍写入 `change_log` 表。但截断条件写的是 `COUNT(change_log) > 1000`，若数据库层面的 `COUNT(*)` 包含 `reindex` 和 `rollback`，则：
- 归一化频繁触发时，`change_log` 中可能积累大量 `reindex` 记录。
- 普通变更可能远未满 1000 条就因总条数（含 reindex）超过 1000 而被截断。

这与「reindex 不计入上限」的意图相矛盾。

**建议:** 明确截断时的计数规则：
> 「截断时仅统计 `type` 为 `create`/`update`/`delete`/`move` 的变更条数；`reindex` 和 `rollback` 类型不计入 1000 条上限，但受 7 天年龄限制（即若 reindex 记录超过 7 天，仍可被年龄条件截断）。」

**涉及段落:** §4.2、§4.4、§7.3。

#### 3.3 回滚操作的 change_log 表达细节

**位置:** §6.5「回滚原子性」、§7.3「rollback 类型」

文档说 rollback 的 `data` 包含 `rollback_to_version` 和 `affected_guids`，并声称「包含完整树差异（通过快照 + 差分）」。但：
- `data` 示例中并未展示完整树差异的具体结构。
- 客户端收到 `rollback` 变更后，是需要重新加载整个快照，还是根据 `data` 中的差异逐项应用？
- 如果回滚目标是 30 天前的版本，而快照保留策略是「最近 500 个或 30 天」，该快照可能已被清理。

**建议:** 补充以下实现约定：
1. `rollback` 变更的 `data` 字段包含**目标快照的完整规范树**（而非差分），或至少包含足够重建树的差异列表。
2. 服务端在执行回滚前必须**确保目标快照存在**；若快照已过期，回滚操作应返回 E005（数据格式错误）或专用错误码，提示用户无法回滚到该版本。
3. 客户端收到 `rollback` 变更时的行为：清空本地相关子树并按目标快照重建（或按差异列表逐项更新）。

**涉及段落:** §6.5、§7.3、§5.1.2。

### P2 — 文档 polish 与长期优化建议

#### 3.4 `SyncRecord` 中 `change_log` 的语义

**位置:** §7.2

```typescript
SyncRecord {
  version: integer
  device_id: string
  change_log: Change[]   // ← 这是本次 sync 产生的变更，还是全局变更日志的引用？
  snapshot_hash: string
  synced_at: integer
}
```

`SyncRecord` 在 §4.2 的 `GlobalState` 中未出现，而在数据模型中却包含 `change_log: Change[]`。若每条 `SyncRecord` 都存储一个 `Change[]` 副本，则 500 个快照/记录 × 500 条变更 = 大量冗余存储。

**建议:** 明确 `SyncRecord` 的 `change_log` 仅存储**本次 sync 请求产生的变更 ID 列表**（`string[]`），而非完整的 `Change` 对象；或直接在 PRD 中说明「`SyncRecord` 为审计记录，不存储完整变更内容，仅引用 `change_log` 中的 `change_id`」。

**涉及段落:** §7.2。

#### 3.5 变更日志「7 天年龄」的计量基准

**位置:** §4.2

文档写「最旧记录年龄 > 7 天」，但未说明是按 `timestamp`（服务端接收时间）还是按 `from_version` 对应的同步时间计算。

**建议:** 明确为「按 `timestamp` 字段计算（服务端接收时间）」。

**涉及段落:** §4.2。

#### 3.6 弱网验收的「10 条变更 < 5 秒」指标偏宽松

**位置:** §12.2

在延迟 200ms + 丢包 5% 条件下，10 条变更的增量同步（先 POST 后 GET）包含 2 次 RTT 和少量数据。5 秒的目标虽然容易达成，但对于用户体验来说并没有挑战性。

**建议:** 可补充一个更严格的可选指标：
- 弱网下首次全量同步（500 书签）< 60 秒
- 或：弱网下增量同步（10 条变更）< 3 秒

这有助于在测试阶段暴露超时重试逻辑的问题。

**涉及段落:** §12.2。

## 5. 结语

PRD v5.0 是一份**成熟、完整、可直接指导开发**的需求文档。从 v1 到 v5，文档经历了系统性的打磨：跨设备 ID 映射、增量同步协议、冲突处理状态机、边界条件、错误恢复、性能与验收指标均已到位。v4 遗留的所有阻塞性问题在 v5 中均已解决。

**建议在进入详细设计前，将以下 3 点写入技术设计附录或实现备忘：**
1. **分页拉取的连续性保证**（避免分页中途 410）；
2. **`reindex`/`rollback` 是否参与 change_log 计数**（建议不参与 1000 条上限统计）；
3. **回滚操作的快照存在性校验与 `rollback` 变更的 `data` 字段详细结构**。

完成上述备忘后，**项目可立即启动 Phase 1 开发**。
