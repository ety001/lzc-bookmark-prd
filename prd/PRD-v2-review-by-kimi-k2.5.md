## 总体评价

PRD v2.0 相比 v1.0 有**显著改进**，核心问题（书签 ID 冲突、增量同步模型、API 幂等性）都已得到妥善解决。文档结构更加完整，技术细节更加清晰，已具备进入开发的条件。

**主要改进：**
- ✅ 双层 ID 映射（GUID + local_id）解决了跨设备书签识别问题
- ✅ 版本号 + 变更日志的增量同步模型清晰可行
- ✅ 删除了 WebSocket，简化为纯 HTTP 轮询
- ✅ API 增加了幂等性设计（client_request_id）
- ✅ 补充了软删除、冲突解决、限流等关键机制

### 4. 核心设计决策 ✅

这是 v2.0 最重要的改进部分。

#### 4.1 书签身份模型 ✅

**双层 ID 映射设计优秀：**
```typescript
// 服务端：GUID 作为主键
Bookmark { guid: string, local_id: string, ... }

// 插件本地：映射表
IdMapping { local_id: string, guid: string, last_synced_version: integer }
```

**优点:**
- 完美解决 Chrome 本地 ID 跨设备冲突问题
- 映射表存储在 storage.local，符合 Manifest V3 要求
- 首次同步返回映射表，流程清晰

**建议:**
- 考虑映射表的版本兼容性（如果插件升级后 GUID 生成规则变化？）

#### 4.2 增量同步模型 ✅

**版本号 + 变更日志设计合理：**
```typescript
GlobalState { current_version: integer, change_log: Change[] }
```

**优点:**
- 基于版本号的增量拉取简单高效
- change_log 保留最近 1000 条，平衡存储与历史需求
- 幂等性通过 change_id 实现

**建议:**
- 明确 change_log 的清理策略（超过 1000 条后如何清理？）
- 考虑 change_log 的持久化存储方式（数据库表结构）

#### 4.3 冲突检测与解决 ✅

**改进显著：**
- 明确了 4 种冲突类型（更新、删除、移动、重命名）
- 删除冲突默认保留修改（防误删），合理
- 冲突解决流程清晰

**建议:**
- "移动冲突"的检测条件可以更明确（同一 GUID，parent_guid 不同）
- 考虑"创建冲突"（两个设备创建了同名书签）是否需要检测

### 6. API 设计 ✅

这是 v2.0 最完善的部分。

#### 6.1 通用规范 ✅

**优点:**
- 统一的成功/错误响应格式
- 完整的错误码表（E001-E007）
- 限流策略明确（每分钟 60/120 次）

**建议:**
- `X-Device-Id` 和 `device_id` 在请求体中重复，是否必要？
- 考虑添加 `X-Request-Id` 用于全链路追踪

#### 6.2 书签接口 ✅

**GET /api/v1/bookmarks** ✅
- `since_version` 参数清晰
- 全量和增量响应格式区分明确
- `id_mappings` 返回设计合理

**POST /api/v1/bookmarks/sync** ✅

**重大改进：**
```json
{
  "base_version": 120,
  "expected_version": 122,
  "client_request_id": "req_uuid_001"
}
```

- `base_version`: 变更基于的版本
- `expected_version`: 乐观锁，防止丢失更新
- `client_request_id`: 幂等性保证

**设计优秀！** 这三个字段共同保证了同步的正确性和幂等性。

**响应设计合理：**
- HTTP 200 返回 conflicts 数组（非错误状态）
- 冲突解决后再提交

**建议:**
- `applied_changes` 返回具体变更 ID 列表，便于调试
- 考虑添加 `server_timestamp` 用于客户端时间校准

#### 6.3 设备接口 ✅

**建议:**
- `POST /api/v1/devices` 的响应未定义，建议返回生成的 device_id 确认

#### 6.4 历史接口 ✅

**优点:**
- 分页参数明确（limit + offset）
- 回滚生成新版本（非原地覆盖），设计合理
