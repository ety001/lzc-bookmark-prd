# PRD 审计报告 (by GLM-5)

**审计对象：** 懒猫书签同步系统 PRD v1.0  
**审计时间：** 2026-04-16  
**审计模型：** GLM-5 (Alibaba)

---

## 一、严重问题（必须修复）

### 1. 书签 ID 跨设备映射缺失

Chrome 在不同设备上为同一书签分配不同的 ID。当前设计直接使用 Chrome 书签 ID 作为唯一标识，这会导致：
- 设备 A 的书签 ID="123" 与设备 B 的书签 ID="456" 无法匹配
- "增量同步"和"冲突检测"完全失效

**修复建议：**
```
数据模型需新增：
- guid: string      // 全局唯一标识符（UUID）
- id_mappings: { local_id: string, guid: string }  // 本地ID与GUID映射表
```

### 2. 首次同步合并策略缺失

场景：设备 A 已有 500 个书签，设备 B 已有 300 个书签（部分重叠），首次同步时如何处理？

当前 PRD 只描述了"全量备份"，没有说明双向合并逻辑。

**修复建议：**
- 新增"首次同步合并策略"章节
- 定义：URL 匹配规则（相同 URL 视为同一书签）
- 定义：文件夹合并规则（同名文件夹合并内容）

### 3. 删除冲突未定义

场景：设备 A 删除书签 X，设备 B 修改书签 X，两者同时同步。

当前冲突检测只考虑"修改冲突"，删除操作会导致数据丢失或状态不一致。

**修复建议：**
```
删除冲突策略：
- 删除优先：任一设备删除，则删除
- 修改优先：保留修改，忽略删除
- 用户选择：弹窗询问（PRD 已有）
```

## 二、重要问题（建议修复）

### 4. 5 秒并发窗口不可靠

"修改时间相差 < 5 秒视为并发"在网络环境不佳时会产生误判（设备 A 修改后网络延迟，设备 B 修改时还未收到 A 的更新）。

**修复建议：**
```
使用版本向量（Version Vector）替代时间戳：
- 每个设备维护自己的版本计数器
- 服务端维护全局版本号
- 冲突检测基于版本号而非时间
```

### 5. WebSocket 用途未明确

架构图显示 WebSocket，但 API 全部是 HTTP。缺少：
- 实时推送的具体场景
- WebSocket 消息格式
- 连接管理（断线重连）

**修复建议：**
要么删除 WebSocket 简化设计，要么补充完整：
```
WebSocket 用途：
- 实时推送其他设备的变更
- 减少轮询开销

消息格式：
{
  "type": "bookmark_changed",
  "device_id": "...",
  "version": 125
}
```

### 6. API 缺少乐观锁

并发同步时可能产生"丢失更新"问题。

**修复建议：**
```
POST /api/v1/bookmarks/sync
Body: {
  "device_id": "...",
  "bookmarks": {...},
  "base_version": 120,  // 已有 ✓
  "expected_version": 120  // 新增：期望当前版本，不匹配则拒绝
}
Response: {
  "success": false,
  "error": "version_conflict",
  "current_version": 125
}
```

## 三、一般问题（可选修复）

### 7. 缺少错误码规范

API 响应格式未定义统一的错误码，客户端难以处理异常。

**建议补充：**
```
错误码规范：
- E001: API Key 无效
- E002: 权限不足
- E003: 版本冲突
- E004: 设备未注册
- E005: 数据格式错误
```

### 8. Rate Limiting 未定义

恶意客户端可能频繁调用 API 导致服务过载。

**建议补充：**
```
限流策略：
- 每设备每分钟最多 60 次同步请求
- 返回 429 状态码 + Retry-After 头
```

### 9. 批量操作 API 缺失

1000+ 书签全量同步时，单个请求体过大可能超时或被拦截。

**建议补充：**
```
批量上传：
POST /api/v1/bookmarks/batch
Body: { "batch_id": "...", "part": 1, "total_parts": 5, "data": [...] }

批量下载：
GET /api/v1/bookmarks?format=batch
返回分块数据流
```

### 10. 开发周期偏乐观

14 天完成包含以下内容：
- Go 后端 + React 前端 + Chrome 插件
- 同步算法 + 冲突处理
- 多设备测试

建议调整为 **18-21 天**，增加 3-5 天 buffer。
