### 3. 系统架构 ⚠️

- 架构图清晰，组件划分合理

#### 3.2 技术约束中的 SQLite/PostgreSQL
- 提到支持两种数据库，但未说明如何切换
- **建议补充:** 通过环境变量配置？还是自动检测？

---

### 4. 功能需求 ⚠️

#### 4.1 懒猫微服

**问题:**
- **4.1.1 API Key 生成**: 未说明是用户手动生成还是自动分配
- **4.1.3 冲突解决**: "提供合并策略" 但未在 API 设计中体现
  - 建议补充 `/api/v1/bookmarks/conflict/resolve` 的完整参数说明

#### 4.2 Chrome 插件

**问题:**
- **4.2.2 增量备份**: "检测变更后只同步增量" 的实现机制不明确

  - **建议明确:** 检测策略（轮询间隔？事件监听？）
- **4.2.4 离线队列**: 未说明队列的持久化方式
  - Chrome storage.local？内存？IndexedDB？

---

### 5. API 设计 ⚠️

**问题:**

#### 5.1 增量同步参数不完整
```
GET /api/v1/bookmarks
Query: since=<timestamp>
```
- `since` 是客户端本地时间还是服务端时间？
- 建议明确时间戳格式（Unix timestamp / ISO 8601）

#### 5.2 书签变更提交缺少幂等性设计
```
POST /api/v1/bookmarks/sync
Body: { "base_version": 120, ... }
```
- 如果网络超时重试，会不会导致重复提交？
- **建议:** 添加 `client_request_id` 字段实现幂等

#### 5.3 缺少错误响应规范
- 所有 API 都未定义错误时的响应格式
- **建议补充:** 统一的错误响应结构，如：
```json
{
  "error": {
    "code": "CONFLICT_DETECTED",
    "message": "书签冲突需要解决",
    "details": { "conflict_id": "xxx" }
  }
}
```

#### 5.4 分页缺失
- `GET /api/v1/bookmarks/history?limit=50` 只有 limit 没有 offset/page
- 历史版本多时如何翻页？

---

### 6. 数据模型 ⚠️

**问题:**

#### 6.1 书签节点 ID 冲突风险
```typescript
id: string  // Chrome 书签 ID
```
- Chrome 书签 ID 是本地生成的，多设备间会重复（如 "1", "2"）
- **严重问题:** 这会导致多设备同步时的 ID 冲突
- **建议:** 使用复合主键 `(device_id, local_id)` 或全局 UUID 映射

#### 6.2 文件夹层级结构未明确
- `children: BookmarkNode[]` 是嵌套结构还是扁平结构？
- 建议明确：存储时用嵌套 JSON 还是关系型表结构（parent_id）

#### 6.3 变更记录缺少冲突标记
```typescript
{
  type: "create" | "update" | "delete"
  // ...
}
```
- 建议添加 `is_conflict: boolean` 和 `resolved_at` 字段

---

### 7. 同步逻辑 ⚠️

**问题:**

#### 7.1 时间窗口策略过于简单
- "修改时间相差 < 5 秒视为并发"
- **问题:** 设备时钟不同步怎么办？
- **建议:** 使用逻辑时钟（version vector）或服务端时间戳

#### 7.2 缺少删除同步的处理
- 流程图和策略都未明确：书签删除如何同步？
- 软删除还是硬删除？删除后恢复？

#### 7.3 首次全量同步的性能问题
- 1000+ 书签的全量 JSON 可能很大
- **建议:** 考虑分页或压缩传输

---

### 8. 安全考虑 ⚠️

**问题:**

#### 8.1 API Key 存储安全
- "API Key 本地加密存储（Chrome storage.sync）"
- **问题:** Chrome storage.sync 本身不加密，只是同步到 Google 账号
- **建议:** 使用 `chrome.storage.local` + 用户密码派生密钥加密

#### 8.2 缺少速率限制说明
- 建议补充 API 的速率限制策略（防止暴力破解 API Key）

---

### 9. 开发计划 ⚠️

**问题:**

#### 9.1 时间估算偏乐观
- Phase 1 的 "LPK 项目创建" 和 "React 管理界面框架" 放在同一天不太现实
- **建议:** 每个任务标注预估工时（小时），便于跟踪

#### 9.2 缺少测试阶段
- 没有专门的测试/QA 阶段
- **建议:** 增加 Phase 5 测试与 Bug 修复（2-3 天）

---

### 10. 风险与依赖 ⚠️

**问题:**

#### 10.1 缺少关键风险
- **数据丢失风险:** 同步错误可能导致书签被误删/覆盖
- **建议:** 增加 "同步前自动本地备份" 机制

#### 10.2 Chrome 政策变更风险
- Manifest V3 对后台服务有限制（如 service worker 5 分钟限制）
- **建议:** 评估 chrome.alarms API 的使用

---

### 11. 验收标准 ⚠️

**问题:**

#### 11.1 性能指标需要细化
- "1000+ 书签同步 < 10 秒" 是在什么网络条件下？
- **建议:** 明确测试环境（本地网络/公网/弱网）

#### 11.2 缺少兼容性验收
- 不同 Chrome 版本的兼容性
- 建议补充最低支持的 Chrome 版本

## 建议补充章节

1. **浏览器兼容性矩阵** - 支持哪些 Chromium 浏览器及版本
2. **数据备份与恢复策略** - 防止同步错误导致数据丢失
3. **错误处理与降级方案** - 网络异常、服务不可用时的行为
4. **隐私与数据所有权声明** - 明确用户数据存储位置和使用方式
5. **性能基准测试方法** - 如何验证 "1000+ 书签 < 10 秒"

---

## 结论

PRD 整体质量良好，架构设计合理，但存在 **2 个 P0 级别问题** 必须解决：

1. **书签 ID 冲突** - 需要重新设计数据模型
2. **API 幂等性** - 需要补充请求 ID 机制

建议在进入开发前修复这些问题，避免后期返工。
