# 懒猫书签同步系统 - PRD v1.0

## 1. 产品概述

### 1.1 产品名称
懒猫书签 (Lazycat Bookmark)

### 1.2 产品定位
基于懒猫微服架构的 Chrome 浏览器书签同步系统，实现多设备间书签的自动备份与同步，类似 Dropbox 的同步盘体验，但专注于浏览器书签。

### 1.3 核心价值
- 跨设备同步：多个 Chrome 浏览器安装插件后，书签自动同步
- 版本历史：保留书签变更历史，支持回滚
- 私有部署：数据存储在用户自己的懒猫微服，不经过第三方
- 增量同步：高效同步，只传输变更数据

---

## 2. 用户场景

### 2.1 典型用户
- 拥有多台电脑的开发人员
- 需要在工作和家庭电脑间同步书签的用户
- 担心浏览器账号同步隐私问题的用户

### 2.2 使用场景

| 场景 | 描述 |
|------|------|
| 首次设置 | 用户在电脑 A 安装插件 → 配置懒猫 API → 全量备份书签 |
| 日常使用 | 用户在电脑 A 添加书签 → 插件自动备份到懒猫 |
| 多设备同步 | 用户在电脑 B 安装插件 → 配置相同 API → 拉取电脑 A 的书签 |
| 冲突处理 | 电脑 A 和 B 同时修改 → 懒猫合并冲突 → 用户确认 |
| 恢复历史 | 误删书签 → 查看历史版本 → 回滚到之前状态 |

## 3. 系统架构

```
┌─────────────────┐         ┌─────────────────┐         ┌─────────────────┐
│  Chrome 插件 A   │         │  懒猫微服 (LPK)  │         │  Chrome 插件 B   │
│  (电脑 1)        │◄───────►│  lzc-bookmark   │◄───────►│  (电脑 2)        │
│                 │         │                 │         │                 │
│ - 书签读取       │  HTTP   │ - Go API 服务    │  HTTP   │ - 书签读取       │
│ - 书签写入       │  WebSocket │ - React 管理界面  │  WebSocket │ - 书签写入       │
│ - 冲突检测       │         │ - 数据存储       │         │ - 冲突检测       │
│ - 本地缓存       │         │ - 版本历史       │         │ - 本地缓存       │
└─────────────────┘         └─────────────────┘         └─────────────────┘
```

### 3.1 组件划分

| 组件 | 技术栈 | 职责 |
|------|--------|------|
| 懒猫微服 | LPK (Go + React) | API 服务、数据存储、冲突处理、Web 管理界面 |
| Chrome 插件 | Manifest V3 | 书签读写、同步触发、UI 展示 |

### 3.2 技术约束

| 约束项 | 要求 | 理由 |
|--------|------|------|
| 懒猫框架 | 必须使用 LPK (Go+React) | 与懒猫微服生态统一 |
| 插件标准 | 必须使用 Manifest V3 | Chrome 官方要求 |
| 通信协议 | 必须 HTTPS | 安全传输 |
| 数据存储 | 支持 SQLite（默认）/ PostgreSQL（可选） | 轻量部署 + 扩展性 |
| 部署方式 | Docker 容器 | 懒猫微服标准 |

## 4. 功能需求

### 4.1 懒猫微服 (lzc-bookmark)

#### 4.1.1 API 鉴权管理
- API Key 生成：用户可生成多个 API Key，用于不同设备
- 权限控制：读/写/删除权限分离
- 设备管理：查看已注册设备，可撤销设备访问

#### 4.1.2 书签存储
- 数据结构：完整保留 Chrome 书签树结构
- 元数据：记录创建时间、修改时间、来源设备
- 版本历史：每次变更保留快照，支持回滚

#### 4.1.3 同步逻辑
- 增量同步：基于时间戳/版本号的增量更新
- 冲突检测：检测同一书签的多设备修改
- 冲突解决：提供合并策略（最新优先/手动选择）

#### 4.1.4 Web 管理界面
- 书签浏览：树形结构展示所有备份的书签
- 历史记录：查看和回滚历史版本
- 设备管理：查看和管理已连接设备

### 4.2 Chrome 浏览器插件

#### 4.2.1 配置管理
- API 配置：懒猫服务地址、API Key
- 同步设置：自动同步间隔、同步触发条件
- 设备标识：自动生成唯一设备 ID

#### 4.2.2 同步功能
- 全量备份：首次使用时备份所有书签
- 增量备份：检测变更后只同步增量
- 拉取同步：从懒猫拉取最新书签合并到本地
- 冲突提示：发现冲突时弹窗提示用户

#### 4.2.3 UI 展示
- 状态图标：工具栏图标显示同步状态
- 弹出面板：查看同步历史、手动触发同步
- 设置页面：完整配置界面

#### 4.2.4 后台服务
- 监听书签变化：使用 chrome.bookmarks API
- 定时同步：按设定间隔自动同步
- 离线队列：网络不可用时排队，恢复后同步

## 5. API 设计

### 5.1 认证接口

```
POST /api/v1/auth/verify
Header: X-API-Key: <key>
Response: { "valid": true, "permissions": ["read", "write"] }
```

### 5.2 书签接口

```
# 获取完整书签树
GET /api/v1/bookmarks
Query: since=<timestamp>  # 增量同步用
Response: { 
  "bookmarks": {...}, 
  "version": 123,
  "synced_at": "2026-04-15T10:00:00Z"
}

# 提交书签变更
POST /api/v1/bookmarks/sync
Body: {
  "device_id": "chrome-abc123",
  "bookmarks": {...},
  "base_version": 120,
  "changes": [...]
}
Response: {
  "success": true,
  "new_version": 125,
  "conflicts": [...]
}

# 解决冲突
POST /api/v1/bookmarks/conflict/resolve
Body: {
  "conflict_id": "...",
  "resolution": "use_local" | "use_remote" | "merge"
}
```

### 5.3 设备接口

```
# 注册设备
POST /api/v1/devices
Body: { "name": "Work Computer", "type": "chrome" }

# 列出设备
GET /api/v1/devices

# 撤销设备
DELETE /api/v1/devices/<device_id>
```

### 5.4 历史接口

```
# 获取历史版本
GET /api/v1/bookmarks/history?limit=50

# 回滚到指定版本
POST /api/v1/bookmarks/rollback
Body: { "version": 100 }
```

## 6. 数据模型

### 6.1 书签节点 (BookmarkNode)

```typescript
{
  id: string          // Chrome 书签 ID
  title: string
  url: string | null  // 文件夹为 null
  parentId: string
  index: integer
  dateAdded: integer
  dateGroupModified: integer
  children: BookmarkNode[]  // 文件夹特有
}
```

### 6.2 同步记录 (SyncRecord)

```typescript
{
  version: integer        // 全局版本号
  device_id: string       // 来源设备
  bookmarks_snapshot: object  // 完整书签树
  changes: Change[]       // 增量变更
  synced_at: timestamp
}
```

### 6.3 变更记录 (Change)

```typescript
{
  type: "create" | "update" | "delete"
  node_id: string
  node_data: object       // 创建/更新时包含
  timestamp: integer
  device_id: string
}
```

### 6.4 冲突记录 (Conflict)

```typescript
{
  conflict_id: string
  node_id: string
  local_version: Change
  remote_version: Change
  created_at: timestamp
  status: "pending" | "resolved"
  resolution: string | null
}
```

## 7. 同步逻辑详解

### 7.1 同步流程

```
┌──────────────┐
│ 插件启动      │
└──────┬───────┘
       │
       ▼
┌──────────────┐     ┌──────────────┐
│ 检查本地缓存  │────►│ 有未同步变更？│───是──► 推送到懒猫
└──────┬───────┘     └──────────────┘
       │ 否
       ▼
┌──────────────┐
│ 请求懒猫版本  │
└──────┬───────┘
       │
       ▼
┌──────────────┐     ┌──────────────┐
│ 版本一致？    │───是─►│ 同步完成      │
└──────┬───────┘     └──────────────┘
       │ 否
       ▼
┌──────────────┐
│ 拉取增量数据  │
└──────┬───────┘
       │
       ▼
┌──────────────┐     ┌──────────────┐
│ 检测冲突？    │───是─►│ 提示用户解决  │
└──────┬───────┘     └──────────────┘
       │ 否
       ▼
┌──────────────┐
│ 合并到本地    │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 同步完成      │
└──────────────┘
```

### 7.2 冲突检测策略

| 检测条件 | 说明 |
|----------|------|
| 同一节点 ID | 两个设备修改了同一个书签 |
| 时间窗口 | 修改时间相差 < 5 秒视为并发 |
| 版本跳跃 | 本地 base_version 落后超过阈值 |

### 7.3 冲突解决策略

1. 自动合并：不冲突的字段自动合并（如只改标题 vs 只改 URL）
2. 最新优先：时间戳最新的覆盖旧的
3. 手动选择：弹窗让用户选择保留哪个版本

## 8. 安全考虑

### 8.1 传输安全
- 强制 HTTPS
- API Key 不在 URL 中传递
- 敏感操作需要二次验证

### 8.2 存储安全
- 书签数据加密存储（可选）
- API Key 本地加密存储（Chrome storage.sync）
- 不记录书签内容到日志

### 8.3 权限控制
- API Key 可设置只读/读写权限
- 设备级权限撤销
- 操作审计日志

## 10. 风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Chrome API 限制 | 书签 API 有速率限制 | 本地缓存 + 批量操作 |
| 懒猫微服稳定性 | 服务不可用导致同步失败 | 离线队列 + 重试机制 |
| 数据冲突复杂 | 多设备并发修改难处理 | 简化策略 + 用户确认 |
| 跨域问题 | 插件访问懒猫 API 受限 | manifest 配置 CORS |
