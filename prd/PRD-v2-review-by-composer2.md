## 1. 总体结论

v2 相对 v1 **明显收敛**：已落实 **GUID + 客户端映射表**、**版本号 + 变更日志**、**删除 WebSocket / 明确轮询**、**错误码与乐观锁字段**、**回滚生成新版本**、**验收指标附测试条件** 等，整体已达到「可进入详细设计与实现」的水准。

仍建议在开发前澄清若干 **数据模型与多端一致性** 细节，并消除文档内部的 **少量自相矛盾**（HTTPS 强制与内网 HTTP 例外、限流维度等）。以下按优先级列出。

**整体评级:** 优秀（相对 v1）  
**工程完备度:** 中高（补齐 P0 后可为高）

## 2. v2 改进亮点（简要）

- **跨设备身份:** §4.1 双层 ID 与客户端 `IdMapping` 方向正确，直接回应 v1 最大缺口。  
- **增量模型:** §4.2 明确「版本 + 变更日志」，与 §6.3.1 `since_version`、§6.3.2 `base_version` / `expected_version` 基本同轨。  
- **冲突与删除:** §4.3、§8.4 覆盖删除 vs 修改，并给出默认策略。  
- **API 规范:** §6.1 统一信封、错误体、错误码表、幂等 `client_request_id`。  
- **安全与运维:** API Key 哈希、审计日志、限流与 `Retry-After`、自签名证书说明更贴近真实部署。  
- **计划与验收:** 20 天拆分更务实；§12.2 性能项附带测试条件，可测性更好。

## 3. 关键问题与建议（按优先级）

### P0 — 实现前建议写进 PRD / 技术设计附录

#### 3.1 服务端 `Bookmark.local_id` 与多设备语义冲突

§4.1、§7.1 在**服务端**书签对象上包含 `local_id`（Chrome 本地 ID）。同一 `guid` 在 **N 台设备**上对应 **N 个不同的 Chrome `local_id`**，若全局书签表每 `guid` 仅一行，则 **单列 `local_id` 无法正确表达多端状态**。

**建议（二选一写死）：**

- **A（推荐）:** 服务端**规范树**仅含 `guid`、`parent_guid`、`index`、`title`、`url`、删除标记与时间戳等；**不写 `local_id`**。各设备的 `local_id ↔ guid` 仅存在于插件 `storage.local`。  
- **B:** 增加关联表 `device_bookmark_map(device_id, guid, local_id, ...)`，一行表示某设备上该 GUID 的本地 ID。

否则易出现「后同步设备覆盖先同步设备的 local_id」或字段语义不清，联调成本集中在后期。

**涉及段落:** §4.1、§6.3.1 `id_mappings`、§7.1。

#### 3.2 变更日志过滤条件需精确到「版本序」而非仅 `base_version`

§4.2 写：返回 `change_log` 中 `base_version > last_known_version` 的变更。§7.3 中 `base_version` 表示「变更基于的版本号」。在多客户端并发下，**仅按 `base_version` 与客户端水位比较** 容易与「已提交的线性版本序列」混淆；晚到的客户端可能需要 **自某一 `applied_version` 起的所有已提交变更**。

**建议:** 为每条变更或每个版本批次定义明确字段，例如：

- `from_version` / `to_version`，或  
- `change_log` 项带 `resulting_version`（该变更被接受后的全局版本），

并在 §4.2 / §6.3.1 中写明 **拉取增量时的排序键与过滤谓词**，避免实现各自解读。

**涉及段落:** §4.2、§6.3.1、§7.2、§7.3。

#### 3.3 Chrome 书签树根与「单根 root」的映射

PRD 中 `parent_guid: "root"` 与示例 `guid: "root"` 简化合理，但 Chrome 侧存在 **书签栏 / 其他书签 /（可选）移动同步根** 等多子树根。需在实现层约定：

- 插件首次全量上传时如何把 Chrome 树 **稳定地挂载**到服务端的 `root` 子树；  
- 是否允许用户只在「书签栏」范围内同步（产品可选），否则全树同步可能违背部分用户预期。

**建议:** 在 §5.2 或附录增加 **「Chrome 根节点映射策略」** 一小段（不必很长）。

**涉及段落:** §7.1、§8.2、§13.1。

### P1 — 强烈建议补充，利于联调与测试

#### 3.4 `base_version` 与 `expected_version` 的联合语义

§6.3.2 同时出现二者，方向正确，但建议用 **一页状态机** 说明：

- 何时返回 **409 / E003**（与「success + conflicts」的边界）；  
- 部分应用变更 +冲突并存时，客户端 **下一步** 必须先 resolve 再 sync，还是允许带冲突继续拉取。

**涉及段落:** §6.1、§6.3.2、§4.3。

#### 3.5 `merge` 与 `merged_data`

§6.3.3 写明 `merge` 需额外 `merged_data`，但无请求示例与字段校验规则（仅标题/URL 可合并？`parent_guid` / `index` 是否允许在 merge 中一并调整？）。

**建议:** 补一个完整 JSON 样例与 **非法 merge 的错误码**（可沿用 E005）。

**涉及段落:** §6.3.3、§5.1.3。

#### 3.6 GET全量响应中的 `id_mappings` 归属

§6.3.1 全量响应含 `id_mappings`（含 `local_id`）。需明确：

- 该列表是 **当前请求设备** 在服务端上次已知映射，还是 **全设备**（通常不应返回全设备）；  
- 新设备首次拉全量时，映射是否 **空**，由插件在应用树时 **本地生成** `local_id` 并仅在后续 sync 中上报。

**涉及段落:** §6.3.1、§4.1「工作流程」步骤3。

#### 3.7 变更日志保留 1000 条与「落后设备」

若某设备长期离线，缺失的变更可能超过 1000 条或快照与日志无法覆盖，需约定：

- 服务端返回 **410 / 专用错误码** 要求客户端 **全量重拉**，或  
- 自动降级为全量树 + 重置 `last_known_version`。

**建议:** 在 §4.2 或 §6.1 错误码中增加一条 **「需全量同步」**。

**涉及段落:** §4.2 `change_log` 保留策略、§8.1。

#### 3.8 限流维度不一致

§6.6 写「每设备每分钟」；§9.4 写「每 API Key 独立限流」。同一 Key 多设备时，二者 **叠加口径** 不同。

**建议:** 统一为「**每 API Key + 每 Device**」或明确只限其一，并在实现与文档中一致。

**涉及段落:** §6.6、§9.4。

### P2 — 文档 polish与产品一致性

#### 3.9 「必须 HTTPS」与「内网 HTTP 模式」

§3.3 写通信必须 HTTPS；§11 风险缓解写「提供 HTTP 模式（仅内网）」。这是 **硬性约束与例外** 关系，应在 §3.3 或 §9.1 **显式写例外条件**（例如仅绑定127.0.0.1、或显式环境变量 `ALLOW_INSECURE_HTTP=true`），避免安全评审与实现各解。

**涉及段落:** §3.3、§11。

#### 3.10 E004「设备未注册」与「首次自动注册」

错误码表写首次同步自动注册，则 **正常路径不应见到 E004**，或 E004 仅用于 **显式调用设备 API** 的失败场景。建议收紧描述，避免读者以为首次 sync 会 400。

**涉及段落:** §6.1 错误码表、§6.4。

#### 3.11 Phase 1 与数据库说明

§3.3 写明 PostgreSQL 可选；§10 Phase 1 仅列 SQLite schema。可接受，但建议在 Phase 1 或附录加一句 **「PG 使用迁移工具且与 SQLite 同 schema」**，避免 Phase 4 才暴露差异。

**涉及段落:** §10、§3.3、§12.4。

#### 3.12 团队书签（§1.4）与 v2 数据模型

§1.4 提团队用户未来规划；正文均为单用户私有命名空间。无需大改，可加 **「v2 数据模型预留 tenant_id（可选）」** 一句或注明 **完全不预留**，以免后续大迁移。

**涉及段落:** §1.4、§7.x。
