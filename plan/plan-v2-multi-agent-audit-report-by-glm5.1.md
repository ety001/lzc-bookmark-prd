## 一、阻断性问题 (Blocker)

### B1. 时间线自相矛盾：标题声称 12 天，实际计划跨度 20 天

**位置:** 标题行 vs 全文时间线

标题：
> **计划周期:** 12 个工作日（约 2.5 周，相比单 Agent 缩短 33%）

实际时间线：
- Phase 1: Day 1-5
- Phase 2: Day 6-9
- Phase 3: Day 10-14
- Phase 4: Day 15-18
- Phase 5: Day 19-20

**Phase 1 和 Phase 2 完全串行（Phase 2 硬依赖 Phase 1），Phase 3 硬依赖 Phase 1+2。** 唯一真正的跨 Phase 并行是 Phase 4（Day 15-18）可以与 Phase 3 后半段重叠，但计划中 Phase 4 仍从 Day 15 开始，与 Phase 3 的 Day 10-14 也是串行的。

实际总工期 = 20 天，与技术方案 v5 的串行计划完全一致。**"缩短 33%"的声明没有依据。**

如果目标是 12 天，需要重新设计并行策略：
- Phase 1 和 Phase 2 在 Day 1-5 并行（Extension Agent 基于 API 定义先行开发）
- Phase 3 在 Day 6-10（Backend 和 Extension 同时开发）
- Phase 4 在 Day 6-12 与 Phase 3 并行
- Phase 5 在 Day 11-12

**影响:** 时间线不可信，无法用于实际排期。

## 二、重要问题 (Major)

### M1. 资源分配表与实际任务不匹配

**位置:** §3.3 资源调度表

| 时间段 | Backend | Extension | Web | 问题 |
|--------|---------|-----------|-----|------|
| Day 6-9 | **1** | 2 | **1** | Backend=1 无对应任务（Phase 3 从 Day 10 开始）；Web=1 无对应任务（Phase 4 从 Day 15 开始） |
| Day 10-14 | 2 | 2 | **1** | Web=1 无对应任务（Phase 4 从 Day 15 开始） |

Day 6-9 和 Day 10-14 期间 Backend=1 和 Web=1 在做什么？计划中没有给出具体任务。

**建议修复:** 如果 Backend 在 Day 6-9 提前开始 Phase 3 的服务端任务（S3-01, S3-03），应明确写入任务分解。如果 Web 在 Day 6-14 做 Phase 4 的准备工作（API 对接分析、组件原型），也应明确。

---

### M2. Phase 3 任务依赖链存在错误

**位置:** §5.1 Phase 3 任务表

```
E3-04 (版本管理) → 前置任务: E3-03
E3-03 (变更应用) → 前置任务: E3-01
E3-01 (先推后拉) → 前置任务: Phase 2, S3-01
```

但 Day 10 调度中写道：
```
Worker-E2: E3-04 版本管理准备
```

E3-04 依赖 E3-03 → E3-01 → S3-01。S3-01 当天才开始（3h 估时），E3-01 要等 S3-01 完成后才能开始，E3-03 要等 E3-01 完成后才能开始。Worker-E2 在 Day 10 无法开始 E3-04，必须等到 Day 11 甚至 Day 12。

**修复:** 调整 Day 10 调度。Worker-E2 在 Day 10 可做 Phase 2 收尾（E2-12 单元测试）或准备 E2→E3 的过渡工作。

---

### M3. Test Agent 在 Phase 1 和 Phase 2 阶段资源为 0

**位置:** §3.3 资源调度表

| 时间段 | Test Agent |
|--------|------------|
| Day 1-5 | **0** |
| Day 6-9 | **0** |
| Day 10-14 | 1 |

但任务分解中：
- S1-11 (Phase 1 单元测试, 4h) 赋给 Test Agent
- E2-12 (Phase 2 单元测试, 3h) 赋给 Test Agent

资源表和任务分配矛盾。如果 Test Agent 在 Day 1-5 不参与，S1-11 谁来做？如果由 Backend Worker 兼做测试，S1-11 的 4h 需要计入 Backend Worker 工时，这会影响其他任务排期。

**修复:** 将 Test Agent 在 Day 1-5 设为 1（或标注由 Backend Worker 兼做），Day 6-9 设为 0.5。

---

### M4. Phase 2 与 Phase 1 的并行策略未兑现

**位置:** §4.2 Phase 间依赖矩阵

| Phase 2 | Phase 1 (API 定义) | 软依赖 | 80% 可并行 |

但实际调度中 Phase 2 完全在 Phase 1 之后（Day 6-9 vs Day 1-5），没有任何并行。所谓的"80% 可并行"和"接口先行策略"（§7.3）在计划中没有被实际使用。

如果要让 Phase 2 真正并行，需要在 Sync-1.2（Day 3 API 定义冻结）后立即启动 Extension Agent，而不是等到 Day 6。

---

### M5. "API 定义冻结"缺少具体产出物

**位置:** §7.2 同步点 Sync-1.2

```
Sync-1.2 (Day 3 18:00): API 定义冻结
  └─ 产出：API 文档 v1.0
  └─ 解锁：Phase 2、Phase 4 可基于此并行
```

但没有对应的前置任务来生成 API 文档。§5.1 的任务列表中没有一个"编写 OpenAPI 文档"的任务。API 文档是从代码注释自动生成还是手动编写？谁来写？什么时候写？

**修复:** 在 S1-05~S1-07 期间增加一个任务"API 文档编写"（约 2h），可由 Doc Agent（Day 1-5 分配 0.5）完成。

---

### M6. 关键路径上的任务估时可能过于乐观

以下关键路径任务估时可能偏低：

| 任务 | 估时 | 风险 |
|------|------|------|
| S1-08 同步 API | 4h | 涉及幂等处理、溢出判断（409 E003）、部分冲突响应、generated_mappings 生成，4h 偏紧 |
| S1-10 冲突检测算法 | 3h | 涉及 5 种冲突类型分类、纯 create 批次检测、子查询批量检测，3h 偏紧 |
| E3-01 先推后拉流程 | 3h | Chrome 三根映射（未在其他任务中体现）+ 先推后拉完整状态机 + 降级策略，3h 远不够 |
| E3-03 变更应用 | 3h | 增量变更应用到本地书签树（create/update/delete/move/reindex/rollback 全类型），3h 偏紧 |

**建议:** 关键路径任务增加 50% 缓冲，或在 Phase 3 增加 1 天（Day 10-15）。

## 三、建议性问题 (Minor)

### m1. Doc Agent 的工作量不明确

资源表显示 Doc Agent 在 Day 1-18 均为 0.5，但任务列表中只在 Phase 5 (Day 19-20) 有具体任务（"发布说明 + 文档整理"）。Day 1-18 期间 Doc Agent 的产出物未定义。

建议明确 Doc Agent 的持续产出：
- Day 3: API 文档 v1.0
- Day 9: 插件用户配置指南
- Day 14: 同步功能说明文档
- Day 18: Web 端操作手册

### m2. 代码冲突解决策略过于简略

§6.3 资源冲突中"代码所有权划分 + 频繁合并"只有一句话。多 Agent 修改同一模块（如 service 层被 S3-01 和 S3-03 同时修改）的具体策略未说明。建议明确：
- 每个文件/目录的 Owner Agent
- 分支策略（feature branch per task？）
- 合并频率（每日？每个同步点？）

### m3. 调度算法伪代码缺乏实用性

§6.1 的 Python 伪代码是一个简化的拓扑排序，但没有考虑：
- Agent 技能匹配（Worker-B1 不能做 Extension 任务）
- 优先级权重
- 时间窗口约束（必须在某同步点前完成）

建议删除或简化为文字描述。

### m4. 缺少风险应急计划

如果关键路径任务延期（如 S1-08 同步 API 比预期多花 1 天），没有明确的应急策略。建议增加：
- 每个 Phase 增加 1 天缓冲
- 关键路径任务的降级方案（哪些功能可以推迟到 v1.1）

### m5. Phase 5 (Day 19-20) 只给 2 天做全量测试+修复，时间过紧

Phase 5 需要完成：Bug 修复 + 性能优化 + 回归测试 + Docker 打包 + .crx 打包 + 发布说明 + 最终验收。2 天（16 工时）对于 5 个 Agent 全员参与勉强够，但没有任何缓冲。如果 Phase 4 遗留的 Bug 较多，Phase 5 一定延期。

## 四、正向反馈

### P1. 三层调度架构设计合理

L1（里程碑）→ L2（Phase 协调）→ L3（任务执行）的分层结构，职责清晰，适合多 Agent 协作场景。

### P2. 同步点设计有针对性

§7.2 的同步点设计（Schema 冻结、API 冻结、集成测试通过）是实用的里程碑检查点，能有效控制并行开发的质量风险。

### P3. Agent 通信协议（附录 A）设计完整

状态同步、依赖通知、风险上报三种消息类型覆盖了多 Agent 协作的主要场景，可直接用于实现。

### P4. 任务卡片模板（附录 B）标准化

每个任务有明确的 ID、负责人、依赖、验收标准、产出物，便于跟踪和管理。

## 七、建议的修订方向

如果要使计划可行，建议以下修订：

**1. 时间线修正（二选一）：**
- 方案 A：承认 20 天工期，删除"12 天"声明，与技术方案 v5 一致
- 方案 B：真正实现并行，在 Day 3 API 冻结后启动 Extension 和 Web，将工期压缩到 14-15 天

**2. 补充缺失任务（约增加 8-10 个任务，~16-20h）：**
- S1-12: 限流中间件 (2h)
- S1-13: 审计日志 (2h)
- S1-14: 健康检查 (1h)
- S1-15: 权重归一化算法 (2h)
- S1-16: 截断 + 快照算法 (3h)
- S1-17: 回滚 API (2h)
- S1-18: 软删除定时任务 (2h)
- S1-19: 冲突解决超时 (1h)
- S1-20: Prometheus 指标 + 告警 (2h)
- E2-13: Chrome 三根映射 (2h)
- E3-07: Service Worker 降级 (2h)

**3. 关键路径增加缓冲：**
- Phase 3 从 5 天增加到 6 天
- Phase 5 从 2 天增加到 3 天

**修订后预估总工期：方案 A 为 22 天，方案 B 为 15-16 天。**

---

**审计结论:** 计划的分层架构和同步点设计值得肯定，但存在两个阻断性问题——时间线自相矛盾（声称 12 天实际 20 天）和技术方案中 45% 的模块没有对应任务。建议补充缺失任务、修正时间线后重新评审。
