# 多 Agent 开发计划审计报告

**审计对象：** `plan/plan-v2-multi-agent.md`  
**对照基线：** `PRD-v6.md` + `tech/TECHNICAL-DESIGN-v5.md`  
**审计人：** Kimi Code  
**审计时间：** 2026-04-17  
**结论：** 开发计划整体结构清晰、并行策略合理，但存在 **4 项严重缺陷**（核心任务遗漏、测试资源严重不足、排期过于乐观、缺少分支策略），**7 项中等风险** 需在启动前修正。

---

## 一、严重缺陷（必须修复）

### 1. 多项核心算法任务在开发计划中完全遗漏
**位置：** Phase 1 (Day 1-5) 和 Phase 3 (Day 10-14) 任务分解

**问题：** 技术方案 v5 中定义了多项关键算法，但开发计划中没有明确的任务分配：

| 技术方案章节 | 算法/功能 | 计划中的对应任务 | 状态 |
|---|---|---|---|
| §6.3 | `NormalizeWeights` 排序权重归一化 | 无 | ❌ 遗漏 |
| §6.4 | `TruncateChangeLog` 变更日志截断 | 无 | ❌ 遗漏 |
| §6.5 | `GenerateSnapshot` 快照生成与清理 | 无 | ❌ 遗漏 |
| §6.6 | `rollback` 服务端回滚 API | 无 | ❌ 遗漏 |
| §4.6 | `GET /bookmarks` 版本溢出处理 (409+E003) | 无 | ❌ 遗漏 |

**影响：** 这些功能是增量同步、版本管理和数据一致性的核心。如果不在计划中显式分配，极易在开发后期发现缺失，导致 Phase 3/4 延期或同步逻辑不完整。

**修复建议：**
- **Phase 1 Day 4-5** 增加任务：
  - `S1-09a`: 快照生成算法 + `snapshots` 表管理（Worker-B2，3h）
  - `S1-09b`: 变更日志截断算法（Worker-B1，3h）
  - `S1-07a`: 排序权重归一化逻辑（Worker-B2，2h）
- **Phase 3 Day 10-11** 增加任务：
  - `S3-05`: 回滚 API 实现（Worker-B1，3h）
  - `S3-06`: `GET /bookmarks` 溢出检测与 409 响应（Worker-B2，2h）

---

### 2. 测试资源投入严重不足且介入过晚
**位置：** §3.3 资源调度、§4.1 Phase 分解

**问题：**
- **Day 1-9 没有专职 Test Agent**：资源调度表显示 Test Agent 在 Day 1-5 和 Day 6-9 均为 0。这意味着 Phase 1（服务端核心）和 Phase 2（插件基础）没有专职测试人员进行接口验收、单元测试审查和早期 Bug 拦截。
- **集成测试时间被极度压缩**：
  - Phase 3 集成测试仅 **Day 14 一天**（包含多设备同步测试）
  - Phase 4 集成测试仅 **Day 18 一天**
  - Phase 5 仅 2 天用于全部 Bug 修复 + 发布

**影响：** 多 Agent 并行开发虽然提升了编码速度，但接口不一致、实现与 PRD 偏离的问题会在集成阶段集中爆发。测试介入过晚 + 集成时间过短，极易导致 "集成地狱"，迫使项目延期或降低发布质量。

**修复建议：**
- **Test Agent 提前到 Day 3 介入**，负责 API 定义冻结后的接口契约测试（Mock/Contract Test）；
- **每个 Phase 预留至少 20% 时间用于集成测试**：
  - Phase 1：Day 5 全天用于集成测试（而非仅半天）
  - Phase 3：Day 13-14 两天用于集成测试
  - Phase 5：从 2 天扩展为 **4 天**（Day 19-22），增加回归测试和稳定性验证缓冲。

---

### 3. 关键路径排期过于乐观，未考虑算法复杂度和调试时间
**位置：** §3.1 里程碑、§4.1 Phase 分解、§5.1 任务估时

**问题：**
1. **Phase 1 Day 5 仅 1 天完成 "同步 API + 版本号管理 + 冲突检测 + 集成测试"**：根据技术方案 v5，冲突检测需要处理 `create_conflict`（同 GUID title+URL 比对）、部分冲突、同批 create 去重、子查询获取最新变更等复杂逻辑。1 天内完成算法实现 + 单元测试 + 集成测试几乎不可能。
2. **Phase 3 仅 5 天（Day 10-14）完成全部同步逻辑**：这包括服务端增量同步 API、分页拉取、冲突检测优化、部分冲突处理、插件先推后拉流程、增量拉取、变更应用、版本管理、冲突检测 UI、解决策略提交、多设备集成测试。这是整个项目技术难度最高的 Phase，5 天对于多 Agent 首次协作来说风险极高。
3. **Phase 5 仅 2 天**：对于首次开发的多 Agent 项目，2 天不足以完成所有 Bug 修复、性能优化、回归测试、Docker 镜像打包、.crx 打包、文档整理和最终验收。

**影响：** 关键路径一旦延误，将直接影响 M3（Day 14）和 M5（Day 20）里程碑，导致整体发布延期。

**修复建议：**
- **Phase 1 延长至 6 天**（Day 1-6），将冲突检测算法独立放在 Day 5-6；
- **Phase 3 延长至 7 天**（Day 10-16），将同步逻辑拆分为 "服务端同步增强"（Day 10-12）和 "插件同步实现 + 冲突 UI"（Day 11-14），集成测试放在 Day 15-16；
- **Phase 5 延长至 4 天**（Day 19-22）。
- **总工期从 12 工作日调整为 16-18 工作日**，仍比单 Agent 的 20 天有优势，但可行性更高。

---

### 4. 缺少 Git 分支策略和 CI/CD 集成说明
**位置：** §6.3 资源冲突解决、附录

**问题：** 计划提到 "代码所有权划分 + 频繁合并" 作为代码冲突的解决策略，但没有说明：
- 使用什么分支模型（Trunk-based、Git Flow、Feature Branch）？
- 多 Agent 同时提交时，CI 流水线如何运行单元测试和集成测试？
- 是否使用 PR/MR 机制进行 Code Review？
- `pnpm workspace` 下，Agent 修改 `shared/` 目录时，如何触发 extension 和 web 的构建验证？

**影响：** 在多 Agent 并行环境下，如果没有明确的代码合并流程和自动化质量门禁，代码冲突和回归问题会迅速积累，最终拖垮集成阶段。

**修复建议：** 在计划中增加一节 "代码管理与 CI 策略"：
```markdown
## 代码管理与 CI 策略
- **分支模型：** Trunk-based Development，所有 Agent 基于 `main` 分支创建短生命周期 feature branch（< 2 天），通过 PR 合并
- **Code Review：** 每个 PR 至少 1 个其他 Agent Review，关键路径算法（冲突检测、版本号管理）需 2 个 Agent Review
- **CI 流水线：**
  - 每次 PR 触发：Go 单元测试、TypeScript 编译、ESLint
  - 每日定时触发：集成测试（docker compose + go test -tags=integration）
  - Phase 3 后每次 PR 触发：Playwright E2E 基础测试
```

## 二、中等风险（建议修复）

### 5. Web 端回滚功能缺少服务端前置任务
**位置：** §4.1 Phase 4 Day 17 任务 `W4-12`

**问题：** Web 管理端的 "回滚功能" 页面需要调用服务端回滚 API（`POST /api/v1/bookmarks/rollback`），但该 API 的开发在 Phase 1 和 Phase 3 的任务列表中完全缺失。这会导致 Phase 4 Day 17 Web Agent 开发回滚页面时，后端 API 尚未实现。

**修复建议：** 在 **Phase 3 Day 10-11** 增加 `S3-05` 回滚 API 实现任务，作为 `W4-12` 的硬依赖。

---

### 6. Service Worker 降级策略任务缺失
**位置：** §4.1 Phase 3

**问题：** 技术方案 v5 和 PRD §5.2.4 要求 Service Worker 休眠唤醒后通过 `chrome.bookmarks.getTree()` diff 检测变更。开发计划中 Phase 3 没有明确分配该任务（只有 `E3-04 版本管理`，不涵盖降级策略）。

**修复建议：** 在 Phase 3 增加任务 `E3-07`：Service Worker 休眠唤醒降级策略（Worker-E2，3h）。

---

### 7. 未预留代码审查时间
**位置：** §6.2 每日调度计划

**问题：** 每日调度计划中，Agent 的工作时间被完全用于编码，没有预留 Code Review 时间。对于多 Agent 项目，Code Review 通常占用每个 Agent 每天 10-20% 的时间。

**修复建议：** 在每日调度中预留固定 Review 窗口（如 17:00-18:00），或明确说明 "估时不包含 Review 时间，实际执行需增加 20% 缓冲"。

---

### 8. `UPDATE ... RETURNING` 兼容性缺少专门验证任务
**位置：** Phase 1 任务分解

**问题：** 技术方案 v5 的 `NextVersion` 严重依赖 `UPDATE ... RETURNING`，而 §11 风险表明确这是 "中概率" 风险点（SQLite 版本兼容性）。但 Phase 1 的任务列表中没有针对该功能的专项测试或兼容性验证任务。

**修复建议：** 在 Phase 1 Day 5 的集成测试中，明确增加验收项：
> "验证 `NextVersion` 在高并发场景下（100 并发请求）无重复版本号、无空洞。"

---

### 9. 数据库 Schema 变更多 Agent 管理策略缺失
**位置：** §7.2 同步点 Sync-1.1

**问题：** Sync-1.1 要求 Day 1 18:00 "数据库 Schema 冻结"，但多 Agent 环境下，如果后续 Phase 发现 Schema 需要微调（如新增索引、调整字段），没有说明如何处理。技术方案 v5 使用 `golang-migrate`，计划中没有说明迁移脚本的责任人和版本命名规则。

**修复建议：** 明确：
> "数据库迁移由 Worker-B1 统一负责。Schema 冻结后，任何修改必须通过新增迁移文件（`YYYYMMDD_HHMMSS_description.up.sql`）实现，不得直接修改已冻结的迁移脚本。"

---

### 10. Phase 2 任务密度过高，容错空间极小
**位置：** §5.1 Phase 2 任务列表

**问题：** Phase 2（Day 6-9，4 天）包含 11 个任务，总估时约 20 小时，平均每天 5 小时编码时间。但插件开发中，Manifest V3 配置、Chrome API 权限申请、UI 组件手写、事件监听、离线队列等任务的实际调试时间往往超过估时（尤其是 Chrome 插件的异步 API 调试）。

**修复建议：** 将 Phase 2 延长至 **5 天**（Day 6-10），或在 Day 9 增加半天缓冲用于联调。

---

### 11. 验收标准中的覆盖率指标可能流于形式
**位置：** §8.2 并行任务验收

**问题：** "单元测试覆盖率 > 80%" 是一个好的目标，但在多 Agent 快速并行开发中，Agent 可能为了赶进度编写低质量测试（如大量无断言测试）来刷覆盖率数字。

**修复建议：** 补充验收维度：
> "关键路径测试必须覆盖：冲突检测（含 create_conflict）、版本号管理（并发场景）、先推后拉流程（含部分冲突）、截断算法（baseline 更新）。"

## 三、值得肯定的设计

| 项 | 评价 |
|---|---|
| **三层架构（L1/L2/L3）** | 职责清晰，适合多 Agent 协作。 |
| **接口先行策略** | Phase 2/4 基于 API 定义并行开发，减少了关键路径阻塞。 |
| **响应不变式表格** | 虽然不在本计划中，但计划明确引用技术方案中的不变式，有助于减少接口歧义。 |
| **资源冲突解决策略** | 提到了优先级队列、时间片轮转、代码所有权划分，考虑较为全面。 |
| **Agent 通信协议模板** | 附录 A/B 提供了标准化的状态同步和任务卡片格式，有助于 Coordinator 统一管理。 |

---

**总结：** `plan-v2-multi-agent.md` 是一份**架构设计优秀的多 Agent 开发计划**，并行策略和协调机制考虑周全。但当前版本存在**核心算法任务大面积遗漏**、**测试资源严重不足**、**关键路径排期过于激进**等问题。如果按当前 12 工作日执行，极有可能在 Phase 3 集成阶段遭遇严重延期。建议在启动前修正 P0 项，将总工期调整至 **16-18 工作日**，并建立明确的代码审查和 CI 门禁机制，以确保多 Agent 并行开发的质量可控。
