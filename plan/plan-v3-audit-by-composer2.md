## 3. 严重问题（建议 P0）

### 3.1 Phase 5：修订承诺、任务量与日历三者冲突

- **修订摘要 P0-6** 写明：「Phase 5 发布 **2 天→4 天**」。  
- **§5.5** Phase 5 列出 P5-01～P5-10，**总估时约 29h**，且含 Bug 修复、CR、回归、Docker、.crx、发布说明、最终验收。  
- **§7.2** 将 Phase 5 压在 **Day 18 当日**（Sync-5.1 14:00、Sync-5.2 18:00），与 **§4.1** 中 Phase 4 的 Day 18「测试 + 发布准备」并行，**无法在半天内消化29h 级工作**。  
- **§3.1** 中 **M4 与 M5 均落在 Day 18**：若 Phase 5 为独立硬阶段，里程碑语义易混淆（「Web 完成」与「发布」同日需强依赖顺序说明）。

**建议：** 二选一并全文统一——- **方案 A：** Phase 5 独占 **Day 19–22（或至少 Day 19–20）**，相应调整「18 个工作日」总工期与 M5 日期；或  
- **方案 B：** 保持 **18 天**，则 **删减 Phase 5 范围**（例如仅发布 checklist + 已测 build），并把 Bug 池/回归大部前移到 Phase 4 末或 Day 17–18 上午，且 **删除修订摘要中「Phase 5 增至 4 天」的表述**。

### 3.2 §4.1 Phase 1 流程图：Sync-1.2 位置时序错误

在 Phase 1 文本框以「Day 5 18:00 Phase 1 集成」收尾后，紧接着出现：

```text
▼ Sync-1.2 (Day 3 18:00): API 定义冻结
```

从读者视角，这表示 **Phase 1 全部结束之后才出现 Day 3 的 API 冻结**，与 **§7.2**（Sync-1.2 在 Day 3）及并行策略 **矛盾**。

**建议：** 将 **Sync-1.2** 画在 Phase 1 **内部**（Day 3 之后分支），或单独增加「Phase 1 时间轴」小图；删除「Day 5 后再接 Day 3」的箭头结构。

## 4. 中等问题（建议 P1）

### 4.1 Phase 1：「S1-11 冲突检测 (2 天)」与单日排期

- **§4.1** 标注 S1-11为 **2 天**。  
- 同一 Phase 的 Day 5 块将 S1-11 与 S1-10 等并列在 **单日**，**§5.1** 中 S1-11 估时为 **6h**。

**建议：** 统一为「**2 人日 / 12h**」或「跨 Day 4–5」，并同步修改 L2 图与 L3 表，避免执行层误解。

### 4.2 Phase 3：T3-03「2 天」与 L3 表 8h

- **§4.1** 写 T3-03「**2 天**」，**§5.3** 为 **8h**。  
若按每天 4h 有效工时，则一致；若「2 天」指日历日全天，则不一致。

**建议：** 改为「**8h（跨 2 个半天）**」或统一用小时。

### 4.3 Day 3 日程与任务依赖的紧凑度

**§6.2 Day 3** 安排 Worker-B1 开始 **S1-09 同步 API**，而 S1-09 依赖 **S1-06、S1-07**。  
**Day 1–2** 的公开粒度较粗，**Day 2** 未单独展开，存在 **S1-05→S1-06** 与 **S1-07** 是否在 Day 3 上午前全部完成的不确定度。

**建议：** 补充 **Day 2** 半日级调度，或在 Day 3 前增加显式 **同步点**（例如「S1-06/S1-07 完成」）。

### 4.4 Phase 5 估时脚注自洽性

**§5.5** 写「29h」与「5 人 × 1 天 × 4h + 额外 = 24h」——**算术与叙事不匹配**；且若 Phase 5 实际为 4 天，脚注需重写。

**建议：** 按最终选定的 Phase 5 **日历天数**重算总工时，并删除矛盾脚注。

### 4.5 E3-05 / S3-05 的日级并行风险

**Day 13** 同一日安排 **S3-05 冲突解决 API** 与 **E3-05 冲突检测 UI**（E3-05 依赖 S3-05）。可行但 **缓冲薄**；若 S3-05 滑窗，UI 日会承压。

**建议：** 在 §6 或风险表中注明 **S3-05 为当日 P0 解锁任务**。

### 4.6 术语「Chrome 三根映射」

全文多次出现 **「三根映射」**。若为笔误（例如「三向/三方/三键 ID」）或业务专有名词，建议在首次出现处 **括注英文或定义**，避免实现歧义。

### 4.7 CI/CD 示例与多仓库/单仓库结构

**§9.3** 示例仅 **Go test + Docker**，未包含 **Extension（npm/vite）** 与 **Web** 流水线。若项目为 monorepo，示例易误导「仅需 Go CI」。

**建议：** 标注「**后端示例，扩展/Web 另 job**」或补最小矩阵（lint + test）。

### 4.8 文末「评审报告」路径

**§861** 指向 `./plan/plan-v2-multi-agent-audit-report-by-*.md` 三份——与当前 `plan/` 下命名方式一致；**若 v3 无独立审计链**，建议改为「v2 审计见…；**v3 以本文档修订摘要为准**」，避免读者以为 v3 已有三份外部评审。
