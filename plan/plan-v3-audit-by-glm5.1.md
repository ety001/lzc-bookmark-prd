## 一、阻断性问题 (Blocker)

### B1. 修订摘要 P0-6 与 Phase 3/5 实际工期严重矛盾

**位置:** 修订摘要第 22 行 vs §4.1 Phase 3/5 实际排期

修订摘要 P0-6 声称：

> Phase 1 冲突检测 1 天→2 天，**Phase 3 同步逻辑 5 天→7 天**，**Phase 5 发布 2 天→4 天**

但正文实际排期：
- **Phase 3: Day 11-14 = 4 天**（不是 7 天）
- **Phase 5: 无独立天数**，被压缩在 Day 18 与 Phase 4 共享（不是 4 天）

这两处是 v2 审计中 B1（工期矛盾）的直接回应，却在 v3 修订摘要中给出了**错误的修复声明**——声称已修正为 7 天和 4 天，实际正文根本没变。读者会以为 v2 的 B1 已修复，但实际上**工期口径仍不一致**。

**影响:** 文档可信度受损。Coordinator 依据修订摘要做决策时会误判项目节奏。

**修复建议:** 二选一：
- **方案 A:** 将 Phase 3 真正扩展到 Day 11-17（7 天），Phase 5 独立 Day 18-21，总工期调整为 21 天
- **方案 B:** 修改修订摘要 P0-6，将 "5→7 天" 改为 "保持 4 天（含算法补充）"，将 "2→4 天" 改为 "合并到 Phase 4 末尾"

### B2. Phase 5 任务量 29h 无法在 Day 18 半天内完成

**位置:** §5.5 Phase 5 任务分解 + §7.2 同步点 + §4.1 Phase 4

Phase 5 包含 P5-01 至 P5-10，**总估时 29h**：
- P5-01~P5-03 Bug 修复各 4h = 12h（可并行）
- P5-04 性能优化 2h（依赖 P5-01）
- P5-05 代码审查 4h（依赖 P5-01~P5-03）
- P5-06 回归测试 4h（依赖 P5-01~P5-05）
- P5-07 Docker 打包 2h（依赖 P5-06）
- P5-08 .crx 打包 1h（依赖 P5-06）
- P5-09 发布说明 2h（依赖 P5-06）
- P5-10 最终验收 2h（依赖 P5-07~P5-09）

**关键路径串行工时:** P5-01(4h) → P5-05(4h) → P5-06(4h) → P5-07/08/09(2h) → P5-10(2h) = **最少 16h 串行**

§7.2 将 Phase 5 安排在 Day 18 同一天：
- Sync-5.1 (Day 18 14:00): 所有 Bug 修复
- Sync-5.2 (Day 18 18:00): v1.0 发布

这意味着 P5-01~P5-03 要在 14:00 前完成（半天），P5-04~P5-10 要在 14:00-18:00（4h）内完成。**29h 的总工作量 + 16h 的关键路径串行，不可能在 1 天（8h 有效工时）内完成。**

另外，Day 18 同时承载 Phase 4 的测试+发布准备（T4-01, T4-02, T4-03），**Worker 资源严重冲突**。

**修复建议:** Phase 5 必须独立占用至少 2 天（Day 19-20），或在各 Phase 末尾前置 Bug 修复和回归，Phase 5 仅保留打包和发布。

### M4. Day 3 的 S1-09 同步 API 依赖未满足

**位置:** §6.2 Day 3 日程

Day 3 安排 Worker-B1 开始 **S1-09 同步 API**（估时 4h）。

§5.1 中 S1-09 的前置任务为 **S1-06 + S1-07**。

- Worker-B1: S1-01 (2h) + S1-02 (3h) = 5h → 下午继续 S1-02 (1h) + S1-05 (3h) = 4h
- S1-05 从 Day 1 14:00 开始，17:00 完成（不含 Code Review）

Day 2 未单独展开。假设 Day 2 继续：
- Worker-B1: S1-06 设备管理（依赖 S1-05，估时 2h）→ Day 2 上午可完成
- Worker-B2: S1-07 书签 CRUD（依赖 S1-03 + S1-04，估时 4h）

**修复建议:** 补充 Day 2 半日级调度，或在 Day 3 前增加 S1-06/S1-07 完成的显式同步点。

### M5. Phase 2 并行策略未兑现——仍为纯串行

**位置:** 修订摘要 P1-8 vs §4.1

### M7. CI/CD 示例仅覆盖 Go 后端，与三端技术栈不匹配

**位置:** §9.3 CI/CD 配置

§9.3 的 GitHub Actions 示例仅包含：
- `go test ./...`
- `docker build`

项目实际包含三端：
- 后端: Go + Gin + SQLite
- Chrome 扩展: TypeScript + Preact + Vite
- Web 管理端: React + Vite + shadcn/ui

示例中**完全没有 Extension 和 Web 的 CI 流水线**（lint、test、build）。如果项目为 monorepo，开发者可能误以为只需配置 Go CI。

**修复建议:** 在 CI 示例中添加 Extension/Web 的 job（至少 `npm run lint && npm run build`），或标注"后端示例，其他端另配 job"。

### M8. Phase 5 脚注算术自相矛盾

**位置:** §5.5 第 400 行

脚注：「**Phase 5 总估时: 29h** (5 人 × 1 天 × 4h/天 + 额外 = 24h，含缓冲)」

2. 但 5 人 × 1 天 × 4h/天 = **20h**（不是 24h）

**修复建议:** 按 Phase 5 实际天数重算，确保脚注中两个数字一致。

## 四、正向反馈

### P1. 修订摘要与 v2 审计形成有效闭环

v3 修订摘要逐条回应了 v2 审计中的 P0/P1 问题（9+17 项），格式清晰，便于追溯。这在多 Agent 项目管理中是优秀实践。

### P2. 工期口径统一为 18 天

消除了 v2 中 12 天/20 天的严重矛盾，M1-M5 里程碑均在 Day 1-18 内，时间线单轴一致。

### P3. 关键缺口大幅补齐

v3 新增了 15 个核心任务（权重归一化、截断算法、快照生成、回滚 API、限流中间件、Chrome 三根映射、Service Worker 降级等），RTM 覆盖率从 ~55% 提升到 ~75%，显著提高了计划的完整性。

### P4. 同步点设计实用

§7.2 的同步点设计（Schema 冻结、API 冻结、各 Phase 集成通过）配合具体的产出物和解锁条件，可直接用于 Coordinator 的每日调度。

### P5. 风险与应急计划超出一般排期文档

§10 的技术风险（SQLite RETURNING、SW 休眠）和进度风险（关键路径延期、Test Agent 不可用）配有具体应急方案和上报时限，体现了工程化思考。
